## 问题清单

### 问题 1：§11.2 依赖表中 env.d.ts 描述与 §2.2/§12 结论矛盾

- **问题描述**：§11.2「各待实现功能的前置依赖」表格第 673 行仍描述 `env.d.ts` 为待实现功能——"新建文件，定义 .vue 模块类型声明"。但 §2.2 明确结论 env.d.ts 在当前项目状态下不造成任何编译或运行时问题，§12 已将 A1 标记为取消（0d）。两处信息矛盾，开发者在阅读 §11.2 时可能误以为 env.d.ts 仍需创建。
- **所在位置**：§11.2 第 673 行 vs §2.2 第 38-43 行 / §12 第 728 行（A1 取消）
- **严重程度**：中 — 纯信息不一致，不导致错误执行，但可能在团队沟通中引发混淆
- **改进建议**：从 §11.2 依赖表中删除 env.d.ts 行，或将其标注为"已取消（Vite 已内置 .vue 类型声明）"

---

### 问题 2：§5.2 声称"JWT 相关逻辑整合在 authStore.ts"与代码事实不符

- **问题描述**：§5.2 第 211 行写道"JWT 相关逻辑全部整合在 authStore.ts 中"。经代码验证，`authStore.ts` 仅将 token 作为字符串存储/读取/清除（`sessionStorage.getItem('token')` / `sessionStorage.setItem('token')`），**不存在任何 JWT 解析逻辑**——无 `atob`、`base64` 解码、payload 提取、`exp` 字段检查。同时，`src/` 下全部 TypeScript 文件中均无 `jwt`/`JWT`/`atob`/`parseToken` 等关键词。报告自身的 §6.2 第 298 行也确认路由守卫"不检测 JWT exp 过期"，与 §5.2 的声称矛盾。
- **所在位置**：§5.2 第 211 行 vs §4.1 authStore 分析（无 JWT 解析函数提及） vs §6.2 第 298 行（确认 JWT 过期检测缺失） vs `authStore.ts` 实际代码（仅字符串存储，无解析逻辑）
- **严重程度**：中 — 影响 A4（useAuth composable）的工作量判断。若 JWT 解析代码不存在（而非已存在仅需抽取），则 A4 实际是从零实现 JWT 解码和过期检测，0.3d 的估算可能偏低（实际约 0.5—0.7d 含 token 格式校验、exp 时钟偏差容忍、边界情况测试）。此外，该错误描述可能让开发者误以为只需重构 authStore 即可获得 JWT 能力。
- **改进建议**：修正 §5.2 描述为"JWT 解析功能完全缺失——authStore 仅将 token 作为不透明字符串管理，未实现 payload 解码和过期检测。需新建 useAuth composable 从零实现 JWT 工具函数"。同步重新评估 A4 的工作量估算。

### 问题 3：A8 修复方案描述不精确，无法按描述实现设计意图

- **问题描述**：§11.2 第 676 行和 §12 A8 行描述最简修复方案为"修改 `router/index.ts:137` 的 `next('/home')` 为 `next(false)` 语义，即：无来源页时回 /home，有来源页时停留在来源页"。但 Vue Router 的 `next(false)` 仅中止导航并保持在当前页面，**不会**在无来源页时回 `/home`。要实现描述的完整语义，需要额外逻辑（如检查 `window.history.length` 或 `from` 参数），而非仅替换调用参数。执行者仅按字面描述替换为 `next(false)` 将遗漏"无来源页回 /home"的降级逻辑。
- **所在位置**：§11.2 第 676 行、§12 第 735 行（A8）
- **严重程度**：低 — 0.1d 的微任务，执行者实际开发时会发现并补全逻辑，但描述不精确降低了"按文档可直接执行"的可操作性
- **改进建议**：明确 A8 的代码变更内容——建议使用 `next(from && from.path !== to.path ? false : '/home')` 或等效逻辑。或在描述中说明"无来源页回 /home"需额外判断。

### 问题 4：C1 文章收藏任务未分析状态管理层需求，可能低估实现范围

- **问题描述**：C1 任务（§12 第 755 行，1.5d）描述为"扩展 useArticleApi.ts + ArticleDetailView toggleCollect 替换占位 + NewsView 收藏状态（可选）+ 收藏列表页（可选）"，仅涉及 composable 和 view 层的变更。但文章收藏功能涉及以下状态管理需求，当前描述均未覆盖：
  1. **收藏按钮的乐观更新**（点击收藏后立即切换 UI 状态，API 失败时回滚）——类似 LifePlan.vue 打卡的乐观更新模式
  2. **`is_collected` 状态跨页面同步**——在 ArticleDetailView 收藏后返回 NewsView，列表卡片上的收藏标记需同步更新
  3. **收藏列表的分页和缓存**——若实现收藏列表页，需要类似 homeStore 的数据管理逻辑
  这些需求的实现位置（新 Pinia Store、扩展现有 Store、或组件内状态）未被分析，而此架构决策直接影响任务范围和工作量。对比 LifePlan.vue 有独立的 lifePlanStore 管理状态，文章收藏若采用同等完整度（独立 Store + 乐观更新 + 缓存），1.5d 的估算可能不足以覆盖。
- **所在位置**：§12 Group C 第 755 行（C1）
- **严重程度**：中 — 不影响前置依赖判断，但可能在实际开发中因范围理解不一致导致工期超预期
- **改进建议**：补充 C1 任务的状态管理架构分析——明确收藏状态的管理层级（建议扩展现有 useArticleApi 为带状态管理的 composable 或新建 articleStore），并基于选定的架构重新确认 1.5d 的估算是否覆盖。可选方案：将收藏列表页明确标记为 Phase 2 范围外以限制 C1 范围。

### 问题 5：搜索方案（C2）未讨论缓存数据时效性问题

- **问题描述**：§11.3 推荐的最简搜索方案为"全量拉取 GET /api/articles（pageSize=100）后在本地进行标题/标签关键词匹配"。报告评估了文章量级增长（>200 条）的扩展性风险，但未评估**缓存时效性**问题：根据 §4.4，`homeStore` 对文章数据实施了 sessionStorage 缓存（TTL 1h），若搜索功能复用此缓存数据，则搜索结果的时效性受限于 1 小时 TTL——新发布的文章在缓存过期前不会出现在搜索结果中。若搜索每次直接请求 API（绕过缓存），则高频搜索会产生额外网络开销。此遗漏在实际使用中可能导致用户搜索结果不包含最新文章。
- **所在位置**：§11.3 第 686-700 行（推荐方案）、§12 C2 第 756 行
- **严重程度**：低 — 文章更新频率不高的场景下影响有限，但若上线后文章频繁发布（如运营活动期），此问题会显著影响搜索体验
- **改进建议**：在 §11.3 推荐方案的适用条件/局限性中补充缓存时效性说明，并建议实现策略（如搜索时始终请求 API 而非使用缓存，或在缓存失效时自动刷新）。

## 整体评价

报告对前端模块的逐项差距分析详尽、后端 API 实现状态验证充分、依赖链分析和并行分组逻辑清晰。v2-v6 多轮修订已解决了大量一致性和可操作性问题。上述 5 个问题中，问题 2（JWT 逻辑不存在于 authStore）是最需关注的实质性错误（直接影响任务估算），问题 1（env.d.ts 依赖表残留）是修订遗漏的清洁工问题，问题 3-5 属于可操作性精度的进一步提升。整体而言，报告已达到可指导开发执行的质量水平，修正上述问题后可直接作为开发排期的决策依据。
