## 当前审查结果

本轮审查报告（b_v1_diag_v1.md）质量质询结果为 LOCATED（诊断结论被确认，不存在导致审查结论不可信的严重问题），共识别 7 个质量问题：

1. **§11.2 与 §12 依赖声明不一致（中）**：B1（注册表单）在 §11.2 标记依赖 useApi.ts + Login.vue 现有结构，但在 §12 Group B 声明 B1→(A2, A4)，即额外依赖 helpers.ts 和 useAuth.ts。B2（编辑资料）在 §11.2 标记依赖 useUserApi.ts + Profile.vue，但 §12 声明 B2→(A5)，即额外依赖 useUI.ts。注册表单仅需 POST /api/auth/register 后由 authStore 处理 token，无需 A2/A4；编辑资料可直接沿用 Profile.vue 已有的 SweetAlert2 交互模式，无需 A5。两处不一致人为制造了不必要的阻塞依赖。
   - **改进建议**：统一两处依赖声明。B1 仅依赖 useApi.ts + Login.vue 现有结构；B2 将 A5 标记为「非严格依赖」并说明 Profile.vue 已可独立使用 SweetAlert2。

2. **Phase 2 并行人力估算与 3 人约束矛盾（中）**：报告称并行开发最少人力 3 人、Phase 2 工期 1.5d，但 Phase 2 有 8 条并行轨道（B1/B2/B3/C1/C2/D1/D2/D3），按 3 人分配后剩余 D3(0.5d) + C2(0.5d) 无人承担，实际总工期 ≥2.2d。此外 C2 搜索功能的 0.5d 假设无 API 依赖，可能严重低估。
   - **改进建议**：将 Phase 2 工期修正为 ≥2.2d（3人），或将人力增至 4-5 人；为 C2 增加前置条件分析。

3. **D3（useSSE 重构）与 A6（新建 useSSE.ts）任务边界模糊（低~中）**：A6 从 chatStore + Admin.vue 抽取通用 SSE 封装，D3 是 Admin.vue 改调 chatStore.sendAdminMessage。但若 A6 抽取后 chatStore 内部改用 useSSE，D3 可能与 A6 高度重叠甚至成为冗余任务。
   - **改进建议**：明确 A6 产出边界——若 chatStore 内部封装 useSSE，D3 变为「Admin.vue 从内联 SSE 迁移至 chatStore.sendAdminMessage」且工作量可降至 0.1-0.2d。

4. **未验证后端 API 实际实现状态（中）**：报告列出 7 个未覆盖 API 端点并据此规划前端并行开发任务，但未检查这些后端端点是否已实现。若后端未实现，前端开发将阻塞或需要 mock。
   - **改进建议**：增加后端 Express 路由文件的简要验证；在并行分组各任务中补充「若对应后端端点未实现，本项需增加 mock API 开发工作量约 Xd」。

5. **搜索功能（C2）前置条件分析缺失（低~中）**：C2 标记为「无前置依赖」「独立功能」、工作量 0.5d，但完全未分析搜索功能需要什么（搜索入口 UI、结果展示方式、搜索 API）。在不明确设计方案和 API 可用性的情况下，工作量可能严重低估（实际 1.5-2d）。
   - **改进建议**：补充搜索功能设计分析（搜索范围、结果展示、是否需要新 API），基于分析更新工作量。

   - **改进建议**：补充 sessionStorage+BroadcastChannel vs localStorage+pinia-plugin-persistedstate 对比分析；给出明确推荐方向；若无法决策则定义 Phase 2 完成后需收集的数据。

   - **改进建议**：确认路由守卫修复已在 §14 P3 后，可讨论其优先级是否应上调；将 Login.vue 死链接标注为用户可感知缺陷。
