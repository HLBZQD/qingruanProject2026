# 质量审查报告 v4

**审查对象**：`a_v4_diag_v1.md`（前端模块实现差距诊断报告 v4）

**审查维度**：需求响应充分度、事实错误/逻辑矛盾、深度与完整性（侧重内部审议未充分覆盖的维度），并以可操作性视角评估诊断建议的可行性、副作用和优先级合理性。

## 问题清单

### 问题 1：§11.2 行661 依赖描述与 §5.2 结论存在事实矛盾

- **问题描述**：§11.2 依赖表 B1（用户注册）行中声明“**不需要** useAuth.ts（A4）——JWT 解析已由 authStore 完成”，但 §5.2（行211-215）明确结论为“authStore 仅将 token 作为不透明字符串（opaque string）存储在 sessionStorage……代码中不存在任何 JWT 解析逻辑：无 atob 解码、无 base64 解码、无 payload 提取、无 exp 字段检查”。两处描述直接矛盾。
- **所在位置**：§11.2 行661 vs §5.2 行211-215
- **严重程度**：高
- **改进建议**：将行661的“JWT 解析已由 authStore 完成”修改为“注册操作仅需存储后端返回的 token/role/user，不涉及 JWT 解析”——这与 §5.2 的事实一致，且该结论（B1 不依赖 A4）本身是正确的，仅表述有误。

### 问题 2：§12 人力与工期估算存在多处数据不一致

- **问题描述**：
  1. 主分配表（行853-857）显示 3 人方案中人2 累计 2.3d 为最长路径，人3（含C2）累计 2.0d；但行861 声称“C2（1.0d）无法在 3 人内消化”，与分配表中 C2 已分配给 人3 的事实矛盾。
  2. 行864 汇总“3 人方案 Phase 2 ≥2.5d”，但主分配表最长路径仅为 2.3d。2.5d 仅出现在调度说明的替代分配（行860，人3=2.8d）或“C2由人1承担”（行861，人1=2.5d）的假设推演中，与主分配表的结论不匹配。
  3. 行859 称人2 的 2.3d“超过 3 人内最短工期”暗示需调整，但 Phase 持续时间由最长路径决定——人2 的 2.3d 本身就是 Phase 2 的工期，不存在“超过”关系。此表述混淆了单人负荷与阶段工期两个概念。
- **所在位置**：§12 人力与工期估算段（行851-867）
- **严重程度**：高
- **改进建议**：
  - 明确选定一个推荐分配方案，基于该方案统一计算并汇总 Phase 工期。以主分配表为例，Phase 2 = 2.3d（max(1.5, 2.3, 2.0)），3 人总工期 = 0.5 + 2.3 + 0.1 = 2.9d。
  - 删除行861中与分配表矛盾的“C2 无法在 3 人内消化”表述。
  - 将行859的“超过 3 人内最短工期”改为客观描述，如“人2 的负荷（2.3d）为当前分配方案的最长路径，是 Phase 2 的约束瓶颈”。
  - 若需保留替代分配方案作为参考，应将其标注为“备选方案”并给出对应的独立工期估算，不混入主方案结论。

### 问题 3：D3 迁移任务未分析 Admin.vue 内联 SSE 与 chatStore.sendAdminMessage 的行为等价性

- **问题描述**：D3 任务（行765）要求 Admin.vue 废弃内联的 SSE 解析逻辑并切换至 `chatStore.sendAdminMessage`，但未验证两者在事件处理上的行为等价性。Admin.vue 内联实现了 `parseSSEBuffer`/`readSSEStream`/`dispatchSSEEvent`（`Admin.vue:36-78`），chatStore.sendAdminMessage（`chatStore.ts:597-636`）通过内部 `sendStreamRequest` 统一流水线调用 `useChatApi.sendAdminChatMessage`。两者的 SSE 事件分发逻辑可能对 admin 特有的 `SSEAgentMessageEvent`/`SSEAgentThoughtEvent` 等事件类型存在处理差异（如消息构造、状态更新、错误处理策略），报告未做比对验证。执行者若直接替换而未经功能回归，可能引入运行时行为变化。
- **所在位置**：§12 Group D，D3 行765；§8.13 行530
- **严重程度**：中
- **改进建议**：在 D3 任务说明中补充行为等价性验证要求：①对比 Admin.vue 与 chatStore 的 SSE 事件分发分支，确认 admin 特有事件类型的处理路径一致；②迁移后至少执行一次 Admin 对话端到端冒烟测试（发送指令→接收 SSE 流式回复→确认消息渲染和状态更新无异常）。

### 问题 4：AI 助手导航（navigate）的 P1 优先级依据不足

- **问题描述**：§14 将“AI 助手导航（navigate）”（行908）列为 P1，优先级定义为“用户可通过界面入口直接感知到功能不可用”。但 `chatStore.ts` 的 `navigate()` 为空函数体（行716-718），仅在 AI 回复含跨模块导航指令时触发。报告未评估以下关键前提：①当前 AI 工作流是否已集成并实际发送导航指令；②若无导航指令，此功能是否在任何用户交互中可被感知。若 AI 尚未集成或未发送导航指令，此功能实际影响为 0，不满足 P1 定义条件。在当前状态下将其与“搜索功能”（首页首屏可交互搜索栏触发“开发中”Toast）并列 P1 可能误导资源优先级。
- **所在位置**：§14 行908
- **严重程度**：中
- **改进建议**：验证 AI 工作流的导航指令集成状态。若 AI 已集成且实际发送导航指令，保持 P1；若 AI 尚未集成导航指令体系，降级为 P2（架构合规项，与 useAuth/useSSE/useUI 补建同级）并备注“待 AI 集成导航指令后升级为 P1”。

### 问题 5：§14 P0 项存在语义重叠——Login.vue 死链接与用户注册实为同一问题的两个方面

- **问题描述**：§14 将“用户注册”（行902，P0）和“Login.vue 死链接「立即注册」”（行903，P0）列为两个独立 P0 项。后者是前者的症状表现——注册表单缺失直接导致“立即注册”链接为自环无反馈链接。将症状和根因列为两个 P0 项可能造成任务分解的混淆：执行者可能将行903的修复（“替换为 Toast 提示或改为非链接样式”）理解为独立于注册功能开发的临时修补任务，而非纳入 B1（注册表单）任务一并解决。若计划以临时 Toast 修补作为注册功能上线前的过渡方案，应在 B1 任务说明中明确此修补将被注册功能整体替换，避免出现“注册功能已上线但死链接修补未回退”的双重维护状态。
- **所在位置**：§14 行902-903
- **严重程度**：低
- **改进建议**：将行902和行903合并为一条 P0 项，描述为“用户注册功能缺失（含 Login.vue 死链接）”，并在说明中标注：若注册功能开发周期较长，可在 B1 中先以临时 Toast 修补死链接问题，注册功能上线后整体替换。

### 问题 6：导航守卫免责声明拒绝行为修复（A8）的最佳路径与最简路径选择缺乏指引

- **问题描述**：A8 任务（行735）同时描述了两条路径：最简方案（仅修改 `router/index.ts:137` 的 `next('/home')` 为来源页保留逻辑，独立于 A5）和最佳实践（等 A5 完成后将 disclaimer 函数移入 useUI）。但报告未给出“选哪条”的明确推荐。两个方案的结果不同：最简方案修复行为但保留架构偏离，最佳实践消除偏离但需等待 A5。执行者可能选择最简方案而长期遗留架构偏离，或选择最佳实践但因 A5 延期导致 A8 阻塞。
- **所在位置**：§12 Group A，A8 行735
- **严重程度**：低
- **改进建议**：给出明确推荐——鉴于 A8 仅 0.1d（最简方案）且与 A5 解耦，建议优先执行最简方案（在 Phase 1 中完成），并在 §13 架构偏离表中标记“免责声明函数位置”偏离项的修复状态为“A8 最简方案已修复行为，架构迁移待 A5 完成后执行”。
