# 质量审查报告 v1

## 审查概述

**审查对象**：`a_v1_diag_v3.md` — 前端模块实现差距诊断报告 v3
**审查轮次**：第 1 次（首轮审查）
**审查维度**：需求响应充分度、事实错误/逻辑矛盾、深度完整性、可操作性

**整体评价**：报告对设计文档与代码的差距逐模块对比，覆盖面广、多数事实判断准确，整体质量中等偏上。但存在若干依赖逻辑不一致、人力估算不可行、后端依赖缺失分析等问题，影响后续并行开发计划的直接执行。

---

## 发现的质量问题

### 问题 1：依赖关系在 §11.2 与 §12 之间不一致

- **问题描述**：§11.2「各待实现功能的前置依赖」表格与 §12「并行开发分组建议」中的依赖声明多处矛盾：
  - §11.2 中「用户注册前端」的依赖为 `useApi.ts, Login.vue 现有结构`，但 §12 Group B 声明 B1→(A2,A4)，即注册需要 `helpers.ts`（日期格式化/防抖截流）和 `useAuth.ts`（JWT 工具）。注册表单仅需调用 `POST /api/auth/register` 后进行自动登录（authStore.login），不需要 A2 的日期格式化、也不需要 A4 作为独立 composable——注册成功后 token 的存储和解析已由 authStore 完成。
  - §11.2 中「编辑资料」的依赖为 `useUserApi.ts (扩展), Profile.vue`，但 §12 Group B 声明 B2→(A5)，即编辑资料依赖 `useUI.ts`。当前 Profile.vue 已直接使用 SweetAlert2 进行 Toast/弹窗交互，编辑资料表单完全可直接沿用此模式，A5 并非硬性前置条件。
- **所在位置**：§11.2 表格第3-4行 vs §12 Group B 第684行（B1→(A2,A4)）、第689行（B2→(A5)）
- **严重程度**：**中** — 人为制造了不必要的阻塞依赖，可能导致开发者等待基础设施补建完毕才能开始功能开发，浪费并行窗口。
- **改进建议**：统一两处的依赖声明。B1 应仅依赖 `useApi.ts`（已存在）和 Login.vue 现有结构，去除 A2/A4；B2 应将 A5 标记为「非严格依赖」或「建议依赖」，并说明 Profile.vue 已可独立使用 SweetAlert2。

### 问题 2：Phase 2 并行人力估算与 3 人约束矛盾

- **问题描述**：报告称「并行开发所需最少人力：3 人」、Phase 2 并行工期 1.5d。但 Phase 2 有 8 条并行轨道（B1/B2/B3/C1/C2/D1/D2/D3），3 人无法在 1.5d 内完成全部 8 条轨道。实际负荷：
  - 人1：C1(1.5d)
  - 人2：B1(1.0d) + D2(0.3d) = 1.3d
  - 人3：D1(1.0d) + B2(0.5d) + B3(0.2d) = 1.7d
  - **剩余**：D3(0.5d) + C2(0.5d) 无人承担，追加到人2或人3后总工期≥2.2d。
  
  此外，C2（搜索功能）的工作量 0.5d 假设无 API 依赖，但实际搜索功能需要后端搜索接口或至少前端本地过滤实现，0.5d 可能严重低估。
- **所在位置**：§12 第729-738行（Phase 2 并行规划）及第743行（人力估算）
- **严重程度**：**中** — 计划本身不可执行，若按此估算排期将导致延期。
- **改进建议**：(a) 将 Phase 2 工期修正为 ≥2.2d（3人），或将人力增至 4-5 人以维持 1.5d 工期；(b) 为 C2 增加前置条件分析——确认是否需要搜索 API 端点，若端点未实现需标记依赖或增加工作量；(c) D3 与 A6 的边界需厘清（见问题3），厘清后 D3 可能缩减为简单的一行代码改动。

### 问题 3：D3（useSSE 重构）与 A6（新建 useSSE.ts）任务边界模糊

- **问题描述**：A6 的任务是「从 chatStore + Admin.vue 抽取通用 SSE 封装」创建 `useSSE.ts` composable；D3 的任务是「Admin.vue 改调 chatStore.sendAdminMessage 统一 SSE 逻辑」。存在两处模糊：
  1. 如果 A6 已将 SSE 逻辑抽取为独立 composable，chatStore 的 `sendAdminMessage` 是否会内部改用 useSSE？D3 的「改调 chatStore.sendAdminMessage」与 A6 的「抽取通用 SSE 封装」之间的关系未定义——Admin.vue 应直接调用 useSSE 还是通过 chatStore 间接调用？
  2. D3 的 0.5d 工作量可能与 A6 高度重叠：A6 抽取逻辑时自然应包含将 Admin.vue 改用统一封装的步骤，如果 D3 是 A6 的一部分，则 D3 是冗余任务。
- **所在位置**：§12 Group A（A6，第676行）与 Group D（D3，第705行）
- **严重程度**：**低~中** — 可能导致重复开发或任务冲突。
- **改进建议**：明确 A6 的产出边界：若 A6 产出的 useSSE composable 应由 chatStore 内部使用（chatStore 封装 useSSE 后对外暴露），则 D3 变为「Admin.vue 从内联 SSE 迁移至 chatStore.sendAdminMessage」且工作量可降至 0.1-0.2d；若 A6 产出后各页面直接调用 useSSE 不再经 chatStore，则 D3 应改为「Admin.vue 迁移至 useSSE composable」。

### 问题 4：未验证后端 API 的实际实现状态

- **问题描述**：报告列出了 7 个未覆盖的 API 端点（POST /api/auth/register、PUT /api/user/profile、GET /api/chat/doctor/:id/conversations、POST /api/articles/:id/collect、DELETE /api/articles/:id/collect、GET /api/articles/collections、GET /api/assistant/conversations），并基于此规划了前端并行开发任务（B1 注册、B2 编辑资料、C1 文章收藏、D1 会话历史）。但报告未检查这些后端端点是否已实现。如果后端未实现，前端开发将阻塞或需要 mock 服务器，实际工作量应包含 mock 开发成本；如果后端已实现，前端工作量可相应缩减。
- **所在位置**：§10 API Composable 覆盖率分析（第559-606行）及 §12 各组任务规划
- **严重程度**：**中** — 前端开发计划可能会因后端不可用而阻塞，实际可并行性被高估。
- **改进建议**：(a) 增加对后端 API 实现状态的简要验证（检查 Express 路由文件是否已注册对应端点）；(b) 在并行分组的每个任务中补充备注：「若对应后端端点未实现，本项需增加 mock API 开发工作量约 Xd」。

### 问题 5：搜索功能（C2）前置条件分析缺失

- **问题描述**：报告将搜索功能标记为「无前置依赖」「独立功能」，工作量 0.5d，但完全未分析搜索功能需要什么。从 Home.vue 的 `onSearch()` 占位（弹出「搜索功能开发中」Toast）到实际的搜索实现，至少需要：
  - 搜索入口 UI（当前只有一个触发点）
  - 搜索结果展示（跳转到搜索结果页或在当前页展示？设计文档未定义搜索结果页）
  - 搜索 API（当前后端可能无 `/api/search` 或 `/api/articles?keyword=` 端点）
  
  在不明确设计方案和 API 可用性的情况下，0.5d 有可能严重低估（如果需要新建搜索页面 + 搜索 API composable + 后端配合，实际可能在 1.5-2d）。
- **所在位置**：§11.2 第636行（搜索功能 → 依赖 "—"）及 §12 Group C 第697行（C2 搜索功能）
- **严重程度**：**低~中** — 搜索功能占位只是一个入口点，实际实现范围不清导致工作量估算不可靠。
- **改进建议**：(a) 补充搜索功能的设计分析：搜索范围（全文/标题/标签）、搜索结果展示方式（页面跳转/内联/跳转 NewsView 带搜索参数）、是否需要新 API；(b) 基于分析更新工作量估算。

### 问题 6：E1（持久化方案切换）缺乏决策框架

- **问题描述**：E1 的描述为「评估 sessionStorage+BroadcastChannel 方案是否保留；若切换回设计原方案需...」，工作量 2.0d。但报告未提供任何决策依据：
  - 未明确列出当前方案的实际问题和设计方案的优劣对比矩阵
  - 未说明「A-D 组功能稳定后再评估」中的「稳定」定义（全部功能完成后？Bug 收敛后？）
  - 未评估两种方案的切换成本和回滚路径
  - 2.0d 的工作量拆解模糊（安装插件 + 修改 3 个文件 + 调整事件监听，实际可能仅需 0.5d 编码 + 较多时间用于回归测试）
  
  这使 E1 成为一个「薛定谔的任务」——开发团队将无法判断此项是否需要执行。
- **所在位置**：§12 Group E（第709-716行）
- **严重程度**：**低~中** — 影响长期计划的可执行性，但位于 Phase 3 暂不阻塞 Phase 1-2。
- **改进建议**：(a) 补充 sessionStorage+BroadcastChannel vs localStorage+pinia-plugin-persistedstate 的对比分析（安全隔离、跨标签页体验、Token 覆盖风险等）；(b) 给出明确的推荐方向（保留当前方案并更新设计文档 OR 迁移到设计原方案）；(c) 若暂时无法决策，定义 Phase 2 完成后需要收集的数据（如用户报告的多标签页问题数量）作为决策输入。

### 问题 7：部分差距项未纳入第14节优先级清单

- **问题描述**：报告正文中识别了若干偏离/差距，但未进入第14节的「关键未实现功能清单」统一排序，导致执行者可能遗漏：
  - 路由守卫免责声明拒绝行为偏离（`router/index.ts:137` next('/home') 应改为 next(false) 语义）— 在 §6.3 汇总表中标记为「低严重度」，但仅在 §12 作为 A8 出现，未入 §14 P0-P3 清单
  - Admin.vue 的 SSE 逻辑重复实现 — 分散在 §4.2、§8.13、§13，未作为独立可行动项汇总
  - Login.vue 死链接「立即注册」—— 这是用户可感知的缺陷（点击后停留在同一页面），应作为独立问题标注
- **所在位置**：§14「关键未实现功能清单」（第768-785行）与分散在其他各节的差距项
- **严重程度**：**低** — 不影响诊断准确性，但影响任务分配的完整性。
- **改进建议**：(a) 将免责声明拒绝行为修复提升至 §14 清单（建议 P2）；(b) 将 Admin.vue SSE 重构作为 §14 独立项（纳入 D3，不存在遗漏）；(c) 将 Login.vue 死链接标注为用户可感知缺陷，建议在注册功能实现前先替换为「注册功能开发中」Toast 或将文本改为非链接样式。

## 审查总结

| 问题编号 | 严重度 | 摘要 |
|:--------:|:------:|------|
| 1 | **中** | §11.2 与 §12 依赖声明不一致（B1→A2/A4、B2→A5 无充分依据） |
| 2 | **中** | Phase 2 并行人力估算与 3 人约束矛盾，实际工期被低估 ~30% |
| 3 | 低~中 | D3 与 A6 任务边界模糊，可能导致重复工作 |
| 4 | **中** | 未验证后端 API 实现状态，前端计划可能因后端不可用而阻塞 |
| 5 | 低~中 | 搜索功能前置条件分析缺失，0.5d 工作量可能严重低估 |
| 6 | 低~中 | E1 持久化决策缺乏框架，使 Phase 3 任务无法直接执行 |

报告的逐模块差距对比（§2-§10）事实准确度高，代码引用精确（已抽样验证 main.ts、authStore.ts、Login.vue、ArticleDetailView.vue、chatStore.ts、Profile.vue、Admin.vue、router/index.ts 中的关键引用，均与报告一致）。主要问题集中在 §11-§12 的依赖分析和并行规划部分——这是后续开发者直接依赖的行动指南，其质量直接决定修复效果。
