## 当前审查结果

质询报告结论：**LOCATED** — 所有 11 项问题均已确认，无被质疑项。

### 问题 1：D3 Admin.vue SSE 迁移任务过度简化，遗漏关键重构步骤（严重程度：中）

- **问题描述**：§12 D3（第 765 行）将 Admin.vue SSE 统一描述为"仅需切换调用入口并丢弃内联实现"，但迁移涉及：(a) 模板中消息数据源从本地 `messages` 切换为 `chatStore.conversations`；(b) 本地 `isStreaming` ref 切换为 `chatStore.isStreaming`；(c) `chatStore.conversations` 被多模式共享，Admin.vue 切换后将在同一列表中看到所有模式的对话历史（混入 doctor/assistant 消息）。
- **改进建议**：(1) 补充 D3 任务的完整迁移 checklist（切换 messages→conversations、isStreaming→chatStore.isStreaming、移除本地 SSE 函数和 sendAdminChatMessage 导入）；(2) 评估 chatStore.conversations 多模式共享对 Admin 视图的影响，必要时补充过滤逻辑或讨论 chatStore 按模式隔离消息列表；(3) 重新评估 D3 工作量（当前 0.3d 未含模板绑定改造）

### 问题 2：C1 文章收藏方案A 未明确跨组件状态共享机制（严重程度：低，由中降级）

- **问题描述**：§11.2 第 662 行及 §12 C1 第 754 行推荐方案A——在 useArticleApi.ts 内建 reactive `collectedMap`。若 `collectedMap` 定义在 `export function` 内部则每组件实例独立，跨页面状态同步无法工作。
- **后注**：当前 `useArticleApi.ts` 中已有函数定义于模块顶层的先例模式，正确实现仅需确认 `collectedMap` 定义于 `export function` 体外。此问题降级为低。
- **改进建议**：明确指定模块级单例实现模式，或补充关键代码骨架说明 `collectedMap` 需定义在 composable 函数体外。

### 问题 3：C2 搜索方案前端全量拉取边界条件与后端约束不匹配（严重程度：低）

- **问题描述**：§11.3 推荐方案适用条件设为"文章总量 < 200 条"，但后端 `parsePagination` 将 `pageSize` 上限钳制为 100。单次调用最多返回 100 条，100-200 条区间需 2 次分页拉取。
- **改进建议**：(1) 修正适用条件为 ≤100 条单页全覆盖、100-200 条需 2 页分页拉取；(2) 在 C2 工作量估算中明确是否已包含分页拉取逻辑。

### 问题 4：A7 Markdown 渲染统一化未验证 DOMPurify 配置一致性（严重程度：低）

- **问题描述**：6 个页面内联调用使用 `DOMPurify.sanitize(html)` 无配置参数（默认允许策略），而 `sanitizeHtml()` 使用显式白名单（更严格）。虽因 Markdown 标准输出在白名单内，实际剥离风险极低，但 `ALLOWED_ATTR` 差异可能导致特殊属性（如 `data-*`）被剥离。
- **改进建议**：在 A7 任务中补充 DOMPurify 配置对比 checklist，或建议替换前在每个页面执行一次 DOMPurify 配置专项冒烟测试。

### 问题 5：B1 注册任务 authStore 集成描述不精确（严重程度：低）

- **问题描述**：§11.2 第 661 行"authStore.login 兼容该响应结构"的表述可能导致执行者误调用 `login()` 而非直接调用 `setAuth()`。注册正确实现应为 `authStore.setAuth(data.token, data.role, data.user)`。
- **改进建议**：将措辞替换为"注册成功后直接调用 authStore.setAuth() 完成登录态初始化"，消除"login"一词的歧义。§12 B1 的描述已经准确，可保留不动。

### 问题 6：B1（注册）与 Phase 1 的并行时序关系未明确（严重程度：低）

- **问题描述**：B1 不依赖 Group A，理论上可在 Phase 1 期间并行启动以缩短总工期约 0.4d，但报告未提及此优化可能。
- **改进建议**：在 §12 调度说明中增加一句：B1 不依赖 Group A，若有人力富余可在 Phase 1 期间并行启动，具体由团队根据实际人力决定。

### 问题 7：诊断报告 §4.2 chatStore state 字段命名与代码不一致（严重程度：低）

- **问题描述**：诊断报告 §4.2 设计要求 state 字段表列名为 `messages`，代码实际使用 `conversations`。
- **改进建议**：将字段名修正为 `conversations`，或增加备注说明命名差异。

### 问题 8：§14 优先级排序存在待商榷项（严重程度：中）

- **问题描述**：(a)"编辑资料"列为 P2，但 Profile.vue 菜单入口已渲染，用户点击后收到 Toast 占位提示，符合 P1"用户可感知"定义，评级偏低；(b)"会话历史加载"列为 P1，但当前 DoctorChatView 无"加载历史消息"交互入口，用户不可感知此功能缺失，P1 依据不足。
- **改进建议**：(1) 将"编辑资料"提升为 P1；(2) 补充"会话历史加载"的 P1 依据或降级为 P2，备注升级条件为"DoctorChatView 开发历史消息加载 UI 入口后"。

### 问题 9：Group A 任务编号缺失 A9（严重程度：低）
