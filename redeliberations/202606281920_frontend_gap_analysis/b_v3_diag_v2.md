### 问题 2：§5.2 声称"JWT 相关逻辑整合在 authStore.ts"与代码事实不符

- **问题描述**：§5.2 第 211 行写道"JWT 相关逻辑全部整合在 authStore.ts 中"。经代码验证，`authStore.ts` 仅将 token 作为字符串存储/读取/清除（`sessionStorage.getItem('token')` / `sessionStorage.setItem('token')`），**不存在任何 JWT 解析逻辑**——无 `atob`、`base64` 解码、payload 提取、`exp` 字段检查。同时，`src/` 下全部 TypeScript 文件中均无 `jwt`/`JWT`/`atob`/`parseToken` 等关键词（grep 验证仅命中 JSDoc 注释中的 `@param token - JWT Token` 参数描述，非逻辑代码）。报告自身的 §6.2 第 298 行也确认路由守卫"不检测 JWT exp 过期"，与 §5.2 的声称矛盾。
- **所在位置**：§5.2 第 211 行 vs §4.1 authStore 分析（无 JWT 解析函数提及） vs §6.2 第 298 行（确认 JWT 过期检测缺失） vs `authStore.ts` 实际代码（仅字符串存储，无解析逻辑）
- **严重程度**：中 — 影响 A4（useAuth composable）的工作量判断。若 JWT 解析代码不存在（而非已存在仅需抽取），则 A4 实际是从零实现 JWT 解码和过期检测，0.3d 的估算可能偏低（实际约 0.5—0.7d 含 token 格式校验、exp 时钟偏差容忍、边界情况测试）。此外，该错误描述可能让开发者误以为只需重构 authStore 即可获得 JWT 能力。
- **改进建议**：修正 §5.2 描述为"JWT 解析功能完全缺失——authStore 仅将 token 作为不透明字符串管理，未实现 payload 解码和过期检测。需新建 useAuth composable 从零实现 JWT 工具函数"。同步重新评估 A4 的工作量估算。

### 问题 3：A8 修复方案描述不精确，无法按描述实现设计意图

- **问题描述**：§11.2 第 676 行和 §12 A8 行描述最简修复方案为"修改 `router/index.ts:137` 的 `next('/home')` 为 `next(false)` 语义，即：无来源页时回 /home，有来源页时停留在来源页"。但 Vue Router 的 `next(false)` 仅中止导航并保持在当前页面，**不会**在无来源页时回 `/home`。要实现描述的完整语义，需要额外逻辑（如检查 `window.history.length` 或 `from` 参数），而非仅替换调用参数。执行者仅按字面描述替换为 `next(false)` 将遗漏"无来源页回 /home"的降级逻辑。
- **所在位置**：§11.2 第 676 行、§12 第 735 行（A8）
- **严重程度**：低 — 0.1d 的微任务，执行者实际开发时会发现并补全逻辑，但描述不精确降低了"按文档可直接执行"的可操作性
- **改进建议**：明确 A8 的代码变更内容——建议使用 `next(from && from.path !== to.path ? false : '/home')` 或等效逻辑。或在描述中说明"无来源页回 /home"需额外判断。

### 问题 4：C1 文章收藏任务未分析状态管理层需求，可能低估实现范围

- **问题描述**：C1 任务（§12 第 755 行，1.5d）描述为"扩展 useArticleApi.ts + ArticleDetailView toggleCollect 替换占位 + NewsView 收藏状态（可选）+ 收藏列表页（可选）"，仅涉及 composable 和 view 层的变更。但文章收藏功能涉及以下状态管理需求，当前描述均未覆盖：
  1. **收藏按钮的乐观更新**（点击收藏后立即切换 UI 状态，API 失败时回滚）——类似 LifePlan.vue 打卡的乐观更新模式
  2. **`is_collected` 状态跨页面同步**——在 ArticleDetailView 收藏后返回 NewsView，列表卡片上的收藏标记需同步更新
  3. **收藏列表的分页和缓存**——若实现收藏列表页，需要类似 homeStore 的数据管理逻辑
  这些需求的实现位置（新 Pinia Store、扩展现有 Store、或组件内状态）未被分析，而此架构决策直接影响任务范围和工作量。对比 LifePlan.vue 有独立的 lifePlanStore 管理状态，文章收藏若采用同等完整度（独立 Store + 乐观更新 + 缓存），1.5d 的估算可能不足以覆盖。
- **所在位置**：§12 Group C 第 755 行（C1）
- **严重程度**：中 — 不影响前置依赖判断，但可能在实际开发中因范围理解不一致导致工期超预期
- **改进建议**：补充 C1 任务的状态管理架构分析——明确收藏状态的管理层级（建议扩展现有 useArticleApi 为带状态管理的 composable 或新建 articleStore），并基于选定的架构重新确认 1.5d 的估算是否覆盖。可选方案：将收藏列表页明确标记为 Phase 2 范围外以限制 C1 范围。

### 问题 5：搜索方案（C2）未讨论缓存数据时效性问题

- **问题描述**：§11.3 推荐的最简搜索方案为"全量拉取 GET /api/articles（pageSize=100）后在本地进行标题/标签关键词匹配"。报告评估了文章量级增长（>200 条）的扩展性风险，但未评估**缓存时效性**问题：根据 §4.4，`homeStore` 对文章数据实施了 sessionStorage 缓存（TTL 1h），若搜索功能复用此缓存数据，则搜索结果的时效性受限于 1 小时 TTL——新发布的文章在缓存过期前不会出现在搜索结果中。若搜索每次直接请求 API（绕过缓存），则高频搜索会产生额外网络开销。此遗漏在实际使用中可能导致用户搜索结果不包含最新文章。
- **所在位置**：§11.3 第 686-700 行（推荐方案）、§12 C2 第 756 行
- **严重程度**：低 — 文章更新频率不高的场景下影响有限，但若上线后文章频繁发布（如运营活动期），此问题会显著影响搜索体验
- **改进建议**：在 §11.3 推荐方案的适用条件/局限性中补充缓存时效性说明，并建议实现策略（如搜索时始终请求 API 而非使用缓存，或在缓存失效时自动刷新）。

### 问题 6：§2.1 和 §2.4 的基础设施偏离未纳入 §13 架构偏离汇总表

- **问题描述**：§2.1 识别了 `setupAxiosInterceptors()` 未被显式调用（设计定义 main.ts 中显式调用，实际 useApi.ts 模块顶层自执行）的偏离；§2.4 识别了样式文件路径偏离（设计 `./styles/variables.css` → 实际 `./assets/variables.css`，设计 `./styles/common.css` → 实际 `./styles/animations.css`）。但这两项偏离在 §13「架构偏离汇总」表中均未列出。§13 汇总了 8 项偏离，遗漏了上述 2 项，导致：
  1. 团队无法从 §13 获取完整的偏离全貌，可能忽略需要决策的样式路径对齐问题
  2. `setupAxiosInterceptors` 的偏离是否需要修复（创建函数 vs 维持现状）没有决策指引——当前方案（模块顶层自执行拦截器）在功能上与设计等价，但未评估多实例场景下的副作用（如 SSR 或测试环境可能重复注册拦截器）
- **所在位置**：§2.1 第 27 行、§2.4 第 58-61 行 vs §13 第 875-886 行（表中缺失此两项）
- **严重程度**：低 — 不影响功能，但降低了偏离管理的完整性（§13 应作为偏离决策的唯一入口）
- **改进建议**：将上两项补充到 §13 架构偏离汇总表中，并对每项给出"维持现状并更新设计文档"或"对齐设计"的明确建议。对于 `setupAxiosInterceptors`：建议更新设计文档以反映模块顶层自执行的实际方案（等价且更简洁），无需创建空壳函数。对于样式路径：建议对齐设计路径（将 `assets/variables.css` 移动至 `styles/variables.css`）或更新设计文档以反映实际路径。

### 问题 7：§12 Phase 2 人力分配中 D3 的延迟启动未计入关键路径计算

- **问题描述**：§12 人力与工期估算（第 852-866 行）将 Phase 2 4 人方案的总工期计算为 1.7d（取人 3 的 D1 1.0d + B2 0.5d + B3 0.2d = 1.7d）。但 D 组依赖链（第 769 行）明确规定 D3 建议在 D1 合入后启动（共享 chatStore.ts 修改以避免合并冲突），且方案将 D3 分配给人 2。人 2 的任务链为 B1(1.0d) + D2(0.3d) + D3(0.2d) = 1.5d，但 D3 的 0.2d 必须在 D1 完成（人 3，1.0d）后方可启动。如果人 2 在完成 B1（1.0d）后直接启动 D3，时间线恰好与人 3 完成 D1 的时间重合（均为 1.0d），D3 可以无缝衔接。但若 D1 出现 0.2d 以内的延期（常见于联调），D3 将阻塞人 2 的进度，导致人 2 的实际工期从 1.5d 延长到 1.5d + D1 延期量。当前的"建议在 D1 合入后启动"描述没有给出缓冲策略或降级方案（如同一人承担 D1+D3 以避免等待依赖）。
- **所在位置**：§12 第 857 行（人 2 任务分配）vs 第 769 行（D 组依赖链）
- **严重程度**：低 — 在当前估算精度下可接受，但执行时若 D1 延期，人 2 被迫等待会产生实际排期偏差
- **改进建议**：在人力分配方案中为 D3 增加缓冲说明（如"D3 建议在 D1 合入后启动，若 D1 延期超过 0.2d，D3 降级为与人 3 串行由同一人完成"）。或显式将 D1 和 D3 分配给同一人（但会拉长其个人工期至 1.2d），消除跨人等待依赖。

### 问题 8：§14 优先级清单中"搜索功能"排序与该功能用户可感知程度不匹配

- **问题描述**：§14 将"搜索功能"列为 P2（架构合规/体验优化），依据为"搜索功能在 Home.vue 中的图标被设计文档标记为'装饰性'（设计文档 1.4 节第 2990 行），设计未定义搜索功能页面、路由或 API 端点"。但该优先级排序存在两个层面的不一致：
  1. **用户可感知程度**：Home.vue 确实渲染了搜索栏和搜索图标（`Home.vue:82-93`），用户点击后收到"搜索功能开发中" Toast——这与 P0 定义"用户点击相关入口时无法完成预期操作"完全匹配。对比同表 P2 的"Markdown 渲染统一化"（用户完全不可感知的代码重构），搜索功能对用户体验的影响显著更重。
  2. **与其他 P2 项的比较**："编辑资料"同样被列为 P2，但编辑资料的入口在 Profile.vue 二级菜单中（曝光量低），而搜索栏在首页首屏（曝光量高）。两者用户影响程度不同却被归为同级。
  报告将搜索功能的设计缺失作为降级理由，但设计文档的"装饰性"标注不应掩盖代码中已实现的搜索入口所制造的**用户功能预期**——用户看到搜索栏即预期可搜索，点击无反应构成功能断层。
- **所在位置**：§14 第 906 行（P2 搜索功能） vs P0 优先级定义（第 892-895 行） vs `Home.vue:82-93`（搜索入口代码）
- **严重程度**：一般 — 不影响开发排期（C2 已在 Phase 2 规划中），但优先级标注可能误导排期决策（若团队严格按 P0→P1→P2 顺序分配资源，搜索功能会排到所有 P1 项之后）
- **改进建议**：将搜索功能提升为 P1，理由是"用户可感知的功能缺口（首页搜索入口已有但不可用），但设计文档未定义完整搜索方案，需前置设计决策（A10）"。或保持 P2 但补充说明："虽搜索栏在首页首屏，但因搜索方案尚未设计定型，当前作为非设计合规项列入 P2；建议在 A10 设计决策完成后重新评估优先级。"
