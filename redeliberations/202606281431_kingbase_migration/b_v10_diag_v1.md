## 问题清单

### 问题 1（严重程度：严重）

**问题描述**：适配层改造路径存在关键遗漏 — 方案多处描述逐文件改造路由层后"立即重启应用并自测"（3.5.2 节步骤 4），但未讨论路由文件改造后 `require` 路径变更问题。当前 11 个路由文件使用 `const { db } = require('../db/database')` 获取数据库实例。改造为 adapter 模式后需改为 `const { getAdapter } = require('../db/database'); const adapter = getAdapter()`。在双导出过渡期（步骤 4），已改造的路由文件使用 `getAdapter()`，未改造的路由文件使用 `db`——但每个路由文件内部是**原子使用**的（要么全用旧 `db`，要么全用新 `adapter`），不存在同一文件内混合使用的情况，因此 3.5.2 节描述的"逐文件改造后自测"在路由文件级别是可行的。然而，方案未讨论一种边界情况：**若某个 handler 需要同时从旧模式和 adapter 模式获取数据**（例如改造进行到一半，某个新 handler 依赖另一个尚未改造的模块）——但基于当前代码结构，各路由文件之间无数据库访问的模块间依赖，此边界情况不成立。综合判断，此为**文档完整性缺失**而非实际工程障碍：方案应在 3.5.2 节增加一句明确声明——"各路由文件之间无数据库访问的模块间依赖，逐文件改造不会产生跨文件适配不一致"。

**所在位置**：第 3.5.2 节"Phase 0 过渡策略"步骤 4（文档第 541 行附近）

**改进建议**：在 3.5.2 节步骤 4 描述中增加路由文件间无数据库访问耦合的声明，明确"逐文件改造后自测"的可行性不依赖于其他路由文件的改造状态。

### 问题 2（严重程度：严重）

**问题描述**：事实错误 — 文档引用了一个不存在的文件 `schema.adapter.js`。第 3.6 节（INSERT 操作的返回值和 ID 获取段落）写道："`schema.adapter.js` 方言辅助函数表中不包含 `sql.insertId()`——ID 获取是 adapter 层的职责"。经核实项目实际代码，项目中不存在 `schema.adapter.js` 文件，也从未在方案的任何其他位置提及此文件名。正确的文件引用应为 `sql.js`（`server/db/sql.js`，方言辅助模块）。实现者按此文件名搜索或尝试创建该文件将产生混淆。

**所在位置**：第 3.6 节，INSERT 操作的返回值和 ID 获取段落（文档第 583 行）

**改进建议**：将 `schema.adapter.js` 修正为 `sql.js`。

---

### 问题 3（严重程度：一般）

**问题描述**：事实错误 — `server.js` 行号引用严重错误。第 3.5.1 节末尾的"`db` 导出消费变化确认"段落写道："当前 `server.js`（第 375 行）解构导入了 `{ initDatabase, db }`"。经核实现有代码，`server.js` 文件总共仅 17 行，实际的导入语句在第 2 行（`const { initDatabase, db } = require('./server/db/database');`）。"第 375 行"这个数字与实际情况完全不符，可能源自某个版本差异或复制错误。若实现者按此行号定位代码，将无法找到目标代码。

**所在位置**：第 3.5.1 节末尾，"`db` 导出消费变化确认（v10 新增）"段落（文档第 500 行）

**改进建议**：将"第 375 行"修正为"第 2 行"。

---

### 问题 4（严重程度：一般）

**问题描述**：逻辑矛盾 — database.js 改造后的导出轮廓与过渡期实际导出不一致。第 3.5 节给出了 `database.js` 改造后的最终状态轮廓，导出为 `module.exports = { getAdapter: () => adapter, initDatabase }`。但第 3.5.2 节的 Phase 0 双导出过渡方案要求在步骤 2-4 期间同时导出旧接口 `db` 和新接口 `getAdapter()`：`module.exports = { db: adapter.db, getAdapter: () => adapter, initDatabase }`。第 3.5 节的代码轮廓是"最终状态"而非"过渡状态"，但该节标题为"database.js 改造"且没有标注"此为 Phase 0 完成后的最终状态"。如果实现者以第 3.5 节为权威参考直接实施最终导出状态，将跳过双导出过渡步骤，导致 11 个未改造的路由文件无法启动——这与第 4 轮迭代反馈中已纠正的"database.js 与路由文件改动顺序矛盾"（问题 1）属于同一根因的不同表现形式，表明该矛盾虽已在 3.5.2 节解决，但 3.5 节未同步体现过渡设计。

**所在位置**：第 3.5 节 `database.js` 改造（文档第 400-436 行）与第 3.5.2 节双导出过渡方案（文档第 528-553 行）之间存在不一致

**改进建议**：在第 3.5 节的代码轮廓上方增加注释或说明文字，标注"以下为 Phase 0 完成后的最终导出状态。Phase 0 过渡期间的导出状态见 3.5.2 节"。

---

### 问题 5（严重程度：一般）

**问题描述**：旧 `init_kingbase.sql` 文件处理策略遗漏。项目当前存在 `server/db/init_kingbase.sql` 文件（文件列表已确认）。方案 v14 修订将 KingbaseES 初始化拆分为两个新文件（`init_kingbase_ddl.sql` + `init_kingbase_seed.sql`），第 10.1 节差异分析仍引用旧文件的结构进行比较，第 16 节文件变更清单列出了两个新文件的重写条目。但方案全文未说明对旧 `init_kingbase.sql` 文件的处理方式——是删除、重命名为 `.bak` 归档、还是保留作为参考但不再被代码引用？若实现者保留旧文件，可能在后续维护中被误认为仍是活跃的初始化脚本。

**所在位置**：第 10 节（整体）、第 16 节文件变更清单（文档第 2186-2187 行）

**改进建议**：在第 16 节文件变更清单中新增一条"删除（或归档）`server/db/init_kingbase.sql`"条目；或在第 10 节开头注明旧文件的处理方式。

### 问题 6（严重程度：一般）

**问题描述**：`scripts/` 目录缺失的前置步骤未声明。方案第 16 节文件变更清单列出了约 10 个新建脚本文件（`scripts/migrate-to-kingbase.js`、`scripts/ci-smoke-test.sh`、`scripts/verify-pg-kingbase.js` 等），但经核实项目当前不存在 `scripts/` 目录（`Glob` 搜索 `scripts/*` 返回空结果）。方案未在任何位置说明需要先创建 `scripts/` 目录这一前置步骤。虽此为一个极低成本的步骤，但对于严格按文档逐项操作的实现者，文件创建到不存在的目录将报错。

**所在位置**：第 16 节文件变更清单（文档第 2188-2195 行）

**改进建议**：在第 16 节开头或 Phase 0 步骤中增加一条"创建 `scripts/` 目录（如不存在）"的说明。

### 问题 7（严重程度：轻微）

**问题描述**：第 9.2 节 admin `/execute` 禁用 `sql` 模式的代码片段（文档第 1463-1474 行）使用了 `return error(res, 'UNSUPPORTED', ...)`，但未说明 `error` 函数的来源。经核实项目实际代码（`server/routes/admin.js`），该文件通过 `const { success, error } = require('../utils/response')` 导入响应工具函数。方案此处缺少导入路径说明，虽不影响理解但降低了作为"可直接实施的代码轮廓"的精确性。

**所在位置**：第 9.2 节，admin `/execute` sql 模式禁用代码片段（文档第 1461-1474 行）

**改进建议**：在代码片段上方增加一行注释说明 `error` 函数导入自 `server/utils/response.js`，或与其他 3.6 节的改造说明保持一致（该节已明确说明各路由文件的改造模式）。

### 问题 8（严重程度：轻微）

**问题描述**：文档中多处使用"约 11 个路由文件"、"约 13 个文件"等近似数量表述（第 3.6 节），但在精确的文件变更清单（第 16 节）中具体列出了每个文件。近似表述与精确清单之间的不一致可能让实现者在规划工作量时产生偏差。实际为准：13 个路由文件共计（12 个改造 + 1 个 upload.js 不变），其中 11 个为核心数据库访问文件 + 1 个 index.js（health 端点）+ 1 个 upload.js（不变）。

**所在位置**：第 3.6 节开头（文档第 557 行："约 13 个文件，其中 11 个涉及数据库访问"）

**改进建议**：将"约 13 个文件"改为精确的"13 个文件（12 个需改造 + 1 个不变）"，以保证与第 16 节一致。
