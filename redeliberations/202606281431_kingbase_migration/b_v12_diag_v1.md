## 具体问题

### 问题1：3.4.5节 init() 方法步骤编号重复

- **所在位置**：方案第3.4.5节，DDL文件执行流程（步骤1-7）与种子数据文件执行流程（步骤6-10）
- **严重程度**：轻微
- **问题描述**：DDL执行流程使用步骤编号1-7，种子数据执行流程的步骤从6开始编号（而非重新从1开始），导致本节内存在两个"步骤6"和两个"步骤7"。例如：DDL的步骤6为"DDL失败处理策略"，种子数据的步骤6为"读取 init_kingbase_seed.sql"。当其他章节交叉引用"见3.4.5节步骤6"时，引用存在歧义。
- **改进建议**：将种子数据执行流程的步骤编号重新从1开始（1-5），或使用不同的编号前缀（如"D1-D7"和"S1-S5"）区分两组步骤。同步检查全文中对3.4.5节步骤编号的交叉引用（如11.4节初始化流程图），确保引用明确指向正确的步骤。

### 问题2：优雅关闭缺少 `pool.end()` 超时保护

- **所在位置**：方案第3.5.1节"优雅关闭"段落（第518-540行）
- **严重程度**：轻微
- **问题描述**：优雅关闭流程调用 `await adapter.close()` → `await this.pool.end()`。`pg.Pool.end()` 的文档行为是"等待所有进行中的查询完成后断开所有连接"。若存在长时间运行的查询（如 Dify API 超时导致的事务挂起）或僵死连接，`pool.end()` 可能阻塞数分钟甚至无限挂起，导致进程无法正常退出（被容器编排平台如K8s强制SIGKILL）。方案未讨论此场景的超时处理。
- **改进建议**：在优雅关闭函数中为 `adapter.close()` 增加超时保护——使用 `Promise.race([adapter.close(), timeout(10000)])`，10秒后若未完成则输出警告日志并强制执行 `process.exit(0)`。此超时值应可配置（如 `DB_SHUTDOWN_TIMEOUT_MS` 环境变量）。同时在风险表或运维维度中补充"优雅关闭超时"条目。

### 问题3：`init_kingbase_seed.sql` 内容创建缺少显式指引

- **所在位置**：方案第10.2节种子数据对齐（第1758-1763行）、第16节文件变更清单（第2246行）
- **严重程度**：轻微
- **问题描述**：方案将 KingbaseES 初始化拆分为 DDL 文件（提供了4张关键表的完整DDL示例）和种子数据文件。对于 DDL 文件，方案给出了27项可逐项勾选的实现清单和4张表的完整DDL参考基准。但对于种子数据文件，方案仅说明"内容对齐 seed.sql"和"使用 `__BCRYPT_HASH_PLACEHOLDER__` 占位符"，未给出翻译后的种子数据内容或等效的实现指引。实现者需自行将 `seed.sql` 中约20条 INSERT 语句（管理员、3个医生、4种糖尿病类型、3篇文章）翻译为 KingbaseES 兼容语法，虽工作量小但存在遗漏或翻译错误的风险。
- **改进建议**：在10.2节末尾或16节文件变更清单中增加一句显式说明："`init_kingbase_seed.sql` 的具体内容以 `server/db/seed.sql` 的 INSERT 语句为来源，逐条翻译以下差异点：(1) 列名对齐10.1节差异分析表；(2) 枚举值使用英文（如 `'diet'` 而非 `'饮食'`）；(3) 时间戳默认值由 trigger 改为 `DEFAULT CURRENT_TIMESTAMP`；(4) 密码使用 `__BCRYPT_HASH_PLACEHOLDER__` 占位符。" 此举将翻译步骤从隐式常识转化为显式操作指引。
