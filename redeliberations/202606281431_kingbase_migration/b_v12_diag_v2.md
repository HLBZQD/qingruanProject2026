## 战略决策独立评估

以下对 3 项关键战略决策进行独立深度评估：

### 战略决策 1：驱动选型 -- `pg` vs `pg-promise`

**方案决策**：选用 `pg`（node-postgres）

**独立评估**：

1. **决策逻辑合理性**：方案给出 4 条理由均成立。`pg` 是 `pg-promise` 的底层依赖（选择没有功能损失）、自建适配层不需要高阶查询封装（避免功能冗余）、`pg.Pool` 原生连接池支持、KingbaseES 官方文档推荐 PostgreSQL 兼容驱动为 `pg`。选择逻辑自洽。

2. **潜在风险识别**：
   - KingbaseES V8R6 与 `pg` v8.12 的兼容性**未经实际验证**。方案已通过 Phase 1 前置验证脚本（`scripts/verify-pg-kingbase.js`）和风险表对应条目处理此风险。
   - `pg` 社区对 KingbaseES 特定行为的支持有限（社区聚焦 PostgreSQL），但方案已通过 `pg.types.setTypeParser` 主动控制类型解析行为，降低了对驱动默认行为的依赖。

3. **评估结论**：决策合理，风险可控。前置验证机制若能严格执行，此决策的安全性可充分保障。

### 战略决策 2：事务模型适配 -- 同步→异步

**方案决策**：将 SQLite better-sqlite3 的同步事务模式改造为 async/await 模式，统一 SqliteAdapter 和 KingbaseAdapter 的事务接口

**独立评估**：

1. **决策必要性**：必然选择。KingbaseES 通过 TCP 连接访问，所有数据库操作均为异步（Promise/async-await），不存在同步适配路径。问题不在于"是否改为异步"，而在于"异步改造的完整性和安全性"。

2. **完整性评估**：
   - 方案在第 3.6 节列出 12 个需改造的路由文件和所有需标记 async 的 handler 函数
   - 方案识别了事务内嵌套函数（`insertAdminLog`）的上下文矛盾并给出适配方案（第 8.3 节）
   - 方案识别了内存幂等锁与异步事务间的竞态窗口并给出缓解方案（第 8.5 节 v4 补充）
   - 方案确认项目未引入 `express-async-errors` 并要求所有 handler 使用 `try/catch + next(e)` 模式（第 3.6 节 v4 明确）
   - 方案对异步启动流程（`server.js`）给出完整改造轮廓（第 3.5.1 节）

3. **遗漏风险**：
   - 事务内 DIFY API 调用（`plan.js` 的 `parsePlanOutput()`）在异步改造后仍在事务外执行（方案决策将 Dify 调用移至事务外），这避免了长事务持有数据库连接的问题，是正确的设计。但若将来有需要事务内异步操作（如发送通知）的场景，方案未讨论此模式。
   - 此为远期场景，当前项目范围内不构成遗漏。

4. **评估结论**：决策正确且覆盖完整。异步改造的关键风险点（事务上下文、竞态窗口、错误传播）均已识别并给出缓解措施。

### 战略决策 3：渐进式迁移路径 -- Phase 0/1/2/3

**方案决策**：四阶段渐进式迁移，Phase 0（适配层构建 + SQLite 验证）、Phase 1（KingbaseES 适配 + 双库对比）、Phase 2（生产灰度切换）、Phase 3（可选移除 SQLite）

## 具体问题

### 问题1：3.4.5节 init() 方法步骤编号重复

- **所在位置**：方案第 3.4.5 节，DDL 文件执行流程（步骤 1-7）与种子数据文件执行流程（步骤 6-10）
- **严重程度**：轻微
- **问题描述**：DDL 执行流程使用步骤编号 1-7，种子数据执行流程的步骤从 6 开始编号（而非重新从 1 开始），导致本节内存在两个"步骤 6"和两个"步骤 7"。例如：DDL的步骤 6 为"DDL 失败处理策略（v16 修订）"，种子数据的步骤 6 为"使用 fs.readFileSync 读取 init_kingbase_seed.sql"。当其他章节交叉引用"见 3.4.5 节步骤 6"时，引用存在歧义。
- **改进建议**：将种子数据执行流程的步骤编号重新从 1 开始（1-5），或使用不同的编号前缀（如"D1-D7"和"S1-S5"）区分两组步骤。同步检查全文中对 3.4.5 节步骤编号的交叉引用（如 11.4 节初始化流程图），确保引用明确指向正确的步骤。

### 问题2：优雅关闭缺少 `pool.end()` 超时保护

- **所在位置**：方案第 3.5.1 节"优雅关闭"段落（第 518-540 行）
- **严重程度**：轻微
- **问题描述**：优雅关闭流程调用 `await adapter.close()` → `await this.pool.end()`。`pg.Pool.end()` 的文档行为是"等待所有进行中的查询完成后断开所有连接"。若存在长时间运行的查询（如 Dify API 超时导致的事务挂起）或僵死连接，`pool.end()` 可能阻塞数分钟甚至无限挂起，导致进程无法正常退出（被容器编排平台如 K8s 强制 SIGKILL）。方案未讨论此场景的超时处理。
- **改进建议**：在优雅关闭函数中为 `adapter.close()` 增加超时保护——使用 `Promise.race([adapter.close(), timeout(10000)])`，10 秒后若未完成则输出警告日志并强制执行 `process.exit(0)`。此超时值应可配置（如 `DB_SHUTDOWN_TIMEOUT_MS` 环境变量）。同时在风险表或运维维度中补充"优雅关闭超时"条目。

### 问题3：`init_kingbase_seed.sql` 内容创建缺少显式指引

- **所在位置**：方案第 10.2 节种子数据对齐（第 1758-1763 行）、第 16 节文件变更清单（第 2246 行）
- **严重程度**：轻微
- **问题描述**：方案将 KingbaseES 初始化拆分为 DDL 文件（提供了 4 张关键表的完整 DDL 示例）和种子数据文件。对于 DDL 文件，方案给出了 27 项可逐项勾选的实现清单和 4 张表的完整 DDL 参考基准。但对于种子数据文件，方案仅说明"内容对齐 seed.sql"和"使用 `__BCRYPT_HASH_PLACEHOLDER__` 占位符"，未给出翻译后的种子数据内容或等效的实现指引。实现者需自行将 `seed.sql` 中约 20 条 INSERT 语句（管理员、3 个医生、4 种糖尿病类型、3 篇文章）翻译为 KingbaseES 兼容语法，虽工作量小但存在遗漏或翻译错误的风险。
