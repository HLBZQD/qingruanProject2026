## 问题清单

### 问题 1（严重）：逻辑矛盾——"前端需负责时区转换"与"前端代码无需任何修改"不可同时成立

- **所在位置**：
  - 第 6 节 Phase 0 关键行为变更（第 1001 行）：明确声明"前端需负责时区转换显示"
  - 第 14 节"前端确认"（第 1895 行）：明确声明"前端代码无需任何修改"
- **问题描述**：
  方案决定 Phase 0 将时间戳存储从本地时间（UTC+8）切换为 UTC，数据库中新写入的时间值将相差 8 小时。第 6 节承认此变更并要求"前端需负责时区转换显示"，但第 14 节断言"前端代码无需任何修改"。若前端代码真为零改动，则时间显示将全部偏差 8 小时（在 UTC+8 时区下），这与 Phase 0 验收标准第 6 条"前端页面中显示的时间正确"直接矛盾。

  第 14 节 v11 补充段落将"前端代码零变动"仅归因于 timestamp JSON 响应格式一致性（即 `pg.types.setTypeParser` 保证两个后端下的字段格式相同），但这只解决了格式一致性问题，未解决语义一致性问题——即使格式相同，UTC 存储的 `"06:30:00"` 在无时区转换的情况下会被前端显示为上午 6:30 而非正确的下午 2:30。这是"格式正确但语义错误"的典型案例。
- **严重程度**：严重。此矛盾直接影响 Phase 0 验收标准的可达性——若前端不做时区转换，验收标准第 6 条"前端页面中显示的时间正确"在逻辑上不可能通过；若前端需要做时区转换，则第 14 节的"零改动"声明必须撤回。实现者从不同章节拿到相互矛盾的指令，无法确定正确的实施路径。
- **改进建议**：
  1. 明确撤回第 14 节"前端代码无需任何修改"的绝对化表述，改为"前端 API 调用代码无需修改，但需增加时间显示层的时区转换逻辑（将 UTC 时间字符串转换为用户本地时区显示）"
  2. 在第 6 节 Phase 0 验收标准第 6 条下补充前端时区转换的具体实现要求（如在 `src/utils/` 中增加 `formatUTCTime()` 工具函数，所有显示 datetime 的组件调用此函数）
  3. 在第 16 节文件变更清单中新增前端文件条目（`src/utils/timezone.js` 新建，及引用此函数的 Vue 组件改造）
  4. 若团队决定 Phase 0 期间不做前端时区转换（接受时间显示偏差），则需撤回第 6 节验收标准第 6 条和"前端需负责时区转换显示"表述，并明确标注此偏差为 Phase 0 已知限制（与第 4.2 节 Phase 0 混合时间戳评估中的"有条件接受"逻辑一致）

### 问题 2（严重）：内部矛盾——`init()` 方法的 DDL 执行策略在两个段落中相互矛盾

- **所在位置**：
  - 第 3.4.5 节步骤 6（第 259 行）："在一个事务内（`BEGIN` → 逐条执行 → `COMMIT`）顺序执行所有语句"
  - 第 3.4.5 节"事务内 DDL 兼容性验证"（第 280 行）："DDL 文件使用 `CREATE TABLE IF NOT EXISTS` 保证幂等，**不在事务中执行**"
  - 第 3.4.5 节"推荐"（第 279 行）："将拆分 DDL/种子文件的方案从'备选'提升为'首推方案'"
- **问题描述**：
  第 3.4.5 节 `init()` 方法的步骤 1-7 描述了一个完整的单文件事务内执行流程（将所有 DDL 和种子数据语句包裹在一个事务中顺序执行）。但在同一节的"事务内 DDL 兼容性验证"子节中，方案推荐将 DDL 和种子数据拆分为两个文件，并明确要求 DDL 文件"不在事务中执行"。这两个指令是互斥的——实现者无法同时遵循"在一个事务内执行所有语句"和"DDL 不在事务中执行"。

  方案将"拆分"从备选提升为首推方案后，`init()` 方法的步骤 1-7 描述并未同步更新以反映拆分后的执行流程（如 DDL 文件如何在事务外逐条执行、种子数据文件如何在单独的事务中执行），导致实现者按照步骤 1-7 实施时将得到与推荐方案相反的结果。
- **严重程度**：严重。实现者按照任一描述实施都会与另一描述冲突，无法确定正确的实现路径。若按步骤 1-6（事务内执行 DDL）实施且 KingbaseES V8R6 对 DDL 隐式提交，则初始化在部分失败后留下不可回滚的中间状态；若按推荐方案（DDL 不在事务中）实施，则需重写步骤 1-7 的实现逻辑。
- **改进建议**：
  1. 做出明确决策：在验证 KingbaseES V8R6 事务内 DDL 行为之前，以"拆分方案"为唯一实施路径（而非"首推"但步骤描述未同步）
  2. 将步骤 1-7 的 `init()` 方法描述重写为拆分方案的实现流程：读取 DDL 文件 → 在事务外逐条执行 DDL（`CREATE TABLE IF NOT EXISTS` 保证幂等）→ 检查 users 表是否需要种子数据 → 若需要则读取种子数据文件并在一个新事务中执行
  3. 将"单文件事务内执行"方案降级为备选方案，且仅在 KingbaseES V8R6 事务内 DDL 验证通过后才考虑
  4. 在风险表中补充"若验证后仍采用单文件事务方案但生产环境行为与验证不一致"的风险项

### 问题 3（一般）：`init_kingbase.sql` 重写方案缺少具体产出物，仅靠差异表格难以保证翻译正确性

- **所在位置**：第 10 节（第 1484-1588 行）
- **问题描述**：
  第 10.1 节列出了 22 项 SQLite ↔ KingbaseES DDL 差异，第 10.2 节定义了翻译规则。但方案全文未提供重写后的 `init_kingbase.sql` 实际内容或关键表的完整 DDL 示例。实现者需要根据差异表和翻译规则手工翻译全部 10 张表的 DDL（含列定义、约束、索引、默认值），任何一项遗漏或翻译错误（如 JSONB 默认值的空字符串处理、CHECK 约束枚举值的中英文切换、8 个缺失索引的逐条补充）都可能导致 KingbaseES 初始化后 schema 与 SQLite 不一致，进而引发运行时错误。

  虽然第 10.3 节设计了 CI schema diff 检查作为事后验证，但事后验证不能替代事前给出正确产物——实现者不应在 CI 失败后才能发现翻译错误。
- **严重程度**：一般。手工翻译 22 项差异的出错概率高，且错误仅在连接 KingbaseES 后才能暴露（Phase 1），调试成本较大。
- **改进建议**：
  1. 为关键表（至少 `users`、`life_plans`、`punch_in`、`user_risk_info`）提供完整的 KingbaseES DDL 示例，作为实现者的参考基准
  2. 将差异表转化为可逐项勾选的实现清单（checklist），降低遗漏风险
  3. 在"翻译规则"表中为每个翻译项标注"是否已在 `init.sql` 中验证对应语法"（部分规则如 `DEFAULT CURRENT_TIMESTAMP` 的 SQLite 兼容性已在 4.2 节讨论，但未回链到翻译规则表）

  - 第 12 节（第 1773 行）：提到 `scripts/migrate-reverse-to-sqlite.js` 但仅有框架性描述
  - 第 12.4 节回退方案（第 1035-1039 行）
- **问题描述**：
  方案明确指出"KingbaseES 期间产生的新数据不会自动同步回 SQLite"是一个已知限制，第 12 节声称"完整的回退预案应包含此逆向迁移能力"并提及了 `scripts/migrate-reverse-to-sqlite.js` 脚本框架。但该脚本的具体实现策略完全缺失：

  2. 逆向迁移时的 SERIAL 序列处理（SQLite 无 SERIAL 概念，如何确保逆向迁移后新 INSERT 的 ID 不冲突）未讨论
  3. JSONB → TEXT 的逆向类型转换未给出方案
  4. 增量迁移（仅迁移 Phase 2 期间新增/变更的行 vs 全量回退）的策略选择未讨论
- **严重程度**：一般。回退是生产运维的关键路径。正向迁移有详尽的 SQL 示例和验证策略（12.1 节 7 维度验证），逆向迁移却仅有文件名引用，不满足生产环境的运维完备性要求。

  2. 明确逆向迁移的验证策略（至少覆盖"行数一致"和"时区反向转换抽样验证"两个维度）
  3. 若团队判断逆向迁移为低优先级（Phase 2 稳定运行后不再回退），应在方案中明确声明此判断及接受的风险

### 问题 5（一般）：`pg` 驱动版本与 KingbaseES V8R6 的兼容性缺乏验证声明

- **所在位置**：第 2 节数据库驱动选型（第 32-50 行）
- **问题描述**：
  方案推荐 `pg` v8.12 作为 KingbaseES V8R6 驱动，理由是 KingbaseES"官方文档推荐的 PostgreSQL 兼容驱动即为 `pg`"。但方案未声明是否已在实际的 KingbaseES V8R6 实例上验证过 `pg` v8.12 的连接、查询、事务、连接池等核心功能。KingbaseES 虽是 PostgreSQL 兼容数据库，但在协议层（如认证握手、错误码映射、特定 SQL 函数的行为）可能存在细微差异。若这些差异在实现阶段才被发现，将导致返工。

  类似地，方案中多处技术判断建立在"PostgreSQL 12 行为"的假设上（如 3.4.8 节 OID 值、4.2.1 节隐式类型转换），但 KingbaseES V8R6 虽基于 PostgreSQL 12 内核，其国产化分支可能有特定修改，行为不完全等同于社区版 PostgreSQL 12。
- **严重程度**：一般。不影响方案整体架构，但若驱动兼容性存在问题，Phase 1 实现阶段会遭遇阻塞。
- **改进建议**：
  1. 在第 2 节增加一项前置验证任务："在 KingbaseES V8R6 实例上验证 `pg` v8.12 的基本 CRUD 操作、事务、连接池功能"

  3. 在风险表中新增"pg 驱动与 KingbaseES V8R6 的协议层兼容性未验证"风险项

### 问题 6（一般）：Phase 1 性能基准对比缺少负载测试方法论，验收标准不可量化

- **所在位置**：第 6 节 Phase 1 验收标准第 5 条（第 1026 行）、第 8.2 节（第 1193-1212 行）
- **问题描述**：
  Phase 1 验收标准要求"记录关键端点的查询耗时（P50/P95/P99 响应时间）"和"KingbaseES 下的响应时间应在可接受范围内（建议 < 2 倍 SQLite 基准值）"。但以下关键参数未定义：
  1. 测试时的并发级别未指定（单请求串行测试 vs 多请求并发测试的 P95 值完全不同）
  2. 测试数据量未指定（10 条记录 vs 10000 条记录的查询耗时不在同一量级）
  3. "可接受范围"仅给出 SQLite 相对值（2x），未给出绝对时间上限（如 P95 < 2000ms），在 SQLite 基准值本身极快（如 5ms）时 2x 仅为 10ms 无意义，在 SQLite 基准值较慢（如 800ms）时 2x 变为 1600ms 可能已接近用户体验劣化阈值
  4. `plan.js` 的批量 INSERT 性能对比仅建议使用多行 VALUES 写入，但未给出不同规模（10 条/50 条/100 条方案项）下的性能数据预估，实现者无法判断是否确实需要批量写入优化
- **严重程度**：一般。缺少量化基准的性能验收标准无法真正指导 Phase 1 的通过/失败判定，可能在实际验收时产生争议。
- **改进建议**：
  1. 在 Phase 1 验收标准中补充：测试数据量（建议与生产环境预期数据量同数量级）、并发模型（如 10 并发持续 30 秒）、绝对时间上限（如所有 API 端点 P95 < 2000ms）
  2. 使用 autocannon 或 k6 给出具体的性能测试脚本示例
  3. 在 8.2 节补充批量 INSERT 性能预估（基于网络延迟假设：每轮网络往返 0.5ms，10 条方案项逐条 INSERT = 5ms，多行 VALUES = 0.5ms），帮助实现者判断"是否需要优化"

### 问题 7（轻微）：`node-sql-parser` 解析失败回退到正则匹配时缺乏可观测性，静默行为分歧风险

- **所在位置**：第 3.4.4 节 `execute()` INSERT ID 获取策略（第 209-216 行）
- **问题描述**：
  KingbaseAdapter 的 `execute()` 优先使用 `node-sql-parser` 解析 SQL AST 判断是否为 INSERT 语句，解析失败时回退到正则匹配 `/^\s*INSERT\s+/i.test(sql)`。两个路径对同一条 SQL 的判断结果理论上一致，但在以下边缘情况下可能分歧：(a) SQL 语法极特殊导致 `node-sql-parser` 解析成功但正则误判（如 SQL 中包含注释 `/* INSERT */` 的场景，正则可能误命中）；(b) `node-sql-parser` 版本升级后解析行为变化。

  方案未要求在回退到正则路径时输出日志（warn 级别），运维人员无法知晓生产环境中是否存在"因解析失败而走了回退路径"的 SQL，也无法区分"正常走正则路径"和"解析器异常后走正则路径"这两种根本不同的运行时状态。若 `node-sql-parser` 静默失败而正则恰好也匹配成功，行为虽然正确但掩盖了解析器的潜在 Bug。
- **严重程度**：轻微。不影响功能正确性（在绝大多数场景下两个路径等价），但降低了生产环境的可观测性和问题排查效率。
- **改进建议**：
  1. 当 `node-sql-parser` 解析失败、回退到正则匹配时，使用 `console.warn` 输出日志（含原始 SQL 摘要和解析失败的异常消息）
  2. 考虑在开发/测试环境中将回退路径视为更高级别的告警（如 `console.error`），确保开发阶段能发现所有解析失败场景
  3. 在 3.4.4 节补充此日志策略说明

### 问题 8（轻微）：文档版本号体系造成理解障碍

- **所在位置**：文件名 `a_v9_tech_v3.md`（第 9 轮迭代的技术文档第 3 版）vs 文档标题"技术方案（v13）"（文件内容经历了 13 次修订）
- **问题描述**：
  文件名中的 `v3` 和标题中的 `v13` 使用了两种不同的版本计数体系——前者是"同一迭代轮次内的技术文档版本"，后者是"文档累计修订次数"。虽然第 v11 修订说明 R6 解释了此约定（"文件名中的序号对应第几轮迭代的产出，内容版本号反映该产出经历的修订轮次"），但对于首次接触此文档体系的实现者，在看到文件名 `a_v9_tech_v3.md` 后打开文档却发现标题是 `（v13）`，会产生困惑——文档究竟是第 9 版还是第 13 版？是否需要寻找 `a_v9_tech_v13.md` 或更高版本的文档？

  此外，若同一迭代轮次内存在多个技术文档版本（如 `a_v9_tech_v1.md`、`a_v9_tech_v2.md`），文件名中的 `v3` 无法区分"哪个是最终版本"——实现者需要打开多个文件逐一比较内容版本号（v13）才能确定最新版。
- **严重程度**：轻微。不影响技术实质，但在多人协作场景下增加了沟通成本和误用错误版本的风险。
- **改进建议**：
  1. 在文档开头的元信息区域（标题下方）增加明确的版本声明，例如："**文件标识**：第 9 轮迭代，技术文档第 3 版 | **内容版本**：v13（累计 13 次修订）"
  2. 如有多版本共存，在文件名中体现内容版本号的一致性标记（如 `a_v9_tech_v13.md`），避免两套计数体系并行
  3. 在 `iteration_history.md` 中增加文档版本命名约定说明
