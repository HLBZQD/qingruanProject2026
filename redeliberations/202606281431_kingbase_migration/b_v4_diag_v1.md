## 审查发现

### 问题 1：Phase 0 增量改造的工程可行性缺陷（database.js 与路由文件改动顺序矛盾）

- **所在位置**：方案第 3.5 节（database.js 改造）、第 3.6 节（路由层改动范围）、第 6 节（Phase 0 迁移路径）
- **严重程度**：严重

**问题描述**：

方案在 Phase 0 的迁移路径中描述为："改造 `server/db/database.js`，导出 adapter 实例" -> "逐文件改造路由层，每个文件改造后自测验证"。这是一种增量式的、逐文件推进的策略。

但工程现实是：当前项目全部 11 个路由文件（`auth.js`、`user.js`、`risk.js`、`plan.js`、`punch.js`、`articles.js`、`admin.js`、`assistant.js`、`doctors.js`、`diabetes.js`、`chat.js`）均通过以下方式导入数据库实例：

```javascript
const { db } = require('../db/database');
```

而方案第 3.5 节将 database.js 的导出改为：

```javascript
module.exports = { getAdapter: () => adapter, initDatabase };
```

一旦 database.js 按照方案改造完成，`db` 导出消失，所有尚未改造的路由文件在 `require('../db/database')` 时 `db` 为 `undefined`，应用无法启动。这意味着**不存在"逐文件改造后自测"的可能性** —— database.js 的改动与所有 11 个路由文件的改动必须原子性地同时完成，否则中间任何状态都是不可运行的。

这直接违背了 Phase 0 "不影响现有功能"和"每个文件改造后自测验证"的核心承诺。方案未讨论任何过渡期兼容策略（如同步导出旧 `db` 和新 `getAdapter`，或使用 feature flag 延迟激活 adapter 路径，或先创建 adapter 文件再一次性切换所有路由文件）。

**改进建议**：

方案需在第 3.5 节或第 6 节 Phase 0 中明确给出过渡策略，建议以下方案之一：

(1) **双导出过渡方案（推荐）**：Phase 0 期间 database.js 同时导出旧接口 `db` 和新接口 `getAdapter()`。SqliteAdapter 内部暴露原始的 better-sqlite3 Database 实例作为 `db` 引用（例如通过 `adapter.db` getter），database.js 的 `db` 导出指向该引用。路由文件可逐个从 `const { db }` 迁移到 `const { getAdapter }`，每改完一个文件即可自测验证，未改造的文件继续使用旧 `db` 接口正常工作。Phase 0 验收完成后移除 `db` 导出。

(2) **先建后切方案**：先创建 `server/db/adapter/` 目录下的所有新文件（SqliteAdapter、DatabaseAdapter），这些文件不改变现有 database.js 的导出。然后一次性改造 database.js + 全部 11 个路由文件，作为一个原子提交。此方案放弃了增量自测，但避免了兼容层。

(3) **在 3.5 节末尾增加"Phase 0 过渡兼容策略"子节**，明确过渡期的导出方案、路由文件改造顺序约束、以及如何验证中间状态。

### 问题 2：Phase 0 UTC 转换脚本与 Phase 2 迁移脚本的时区双重转换冲突

- **所在位置**：方案第 4.2 节（Phase 0 混合时间戳处理）、第 12 节（数据迁移时区转换）
- **严重程度**：严重

**问题描述**：

方案在 Phase 0 引入了一个一次性 SQL 脚本 `scripts/phase0_utc_convert.sql`（第 4.2 节），将现有 SQLite 数据中所有 datetime 字段从本地时间（UTC+8）转换为 UTC（减去 8 小时）。执行后，SQLite 数据库进入全 UTC 状态。

随后在 Phase 2 的数据迁移方案中（第 12 节第 5 条），`scripts/migrate-to-kingbase.js` **再次对同样的 datetime 字段减去 8 小时**进行时区转换：

> "现有 SQLite 数据中的 datetime 字段使用 `datetime('now','localtime')` 存储为本地时间（UTC+8），而技术方案决定统一使用 UTC 存储。迁移脚本必须对以下表中所有 datetime 字段进行时区转换（减去 8 小时）"

如果项目按照 Phase 0 -> Phase 1 -> Phase 2 的顺序推进，且 Phase 0 的 UTC 转换脚本已执行，那么 Phase 2 迁移脚本会**再次减去 8 小时**，导致所有时间数据错误偏移（总共减去 16 小时）。例如原始时间 `2025-06-28 14:30:00` (UTC+8) 经过 Phase 0 变为 `2025-06-28 06:30:00` (UTC)，再经过 Phase 2 变为 `2025-06-27 22:30:00`，比正确值提前 8 小时。

方案未讨论这两个脚本之间的互斥关系——它们是二选一的关系（如果 Phase 0 已执行，Phase 2 不应再做时区转换），但方案将它们描述为各自独立的必要步骤。

**改进建议**：

(1) 在 Phase 2 迁移脚本（第 12 节）中增加**时区转换检测逻辑**：检查 SQLite 数据是否已为 UTC（例如检查 `users` 表中最新记录的 `created_at` 值与当前 UTC 时间的差值）。如果数据已是 UTC（即 Phase 0 脚本已执行），则跳过时区转换步骤。

(2) 或者明确**二选一策略**：在方案中声明"Phase 0 UTC 转换脚本与 Phase 2 迁移脚本的时区转换步骤互斥，二者只能执行其一"。推荐选择 Phase 2 统一处理（移除 Phase 0 的独立 UTC 转换脚本），或选择 Phase 0 预处理（Phase 2 迁移脚本检测并跳过时区转换）。

(3) 在第 15 节风险表中增加"Phase 0/Phase 2 时区双重转换"风险项。

### 问题 3：`sql.js` 方言辅助函数的数据库类型感知机制缺失

- **所在位置**：方案第 4.2 节（方言统一策略）
- **严重程度**：一般

**问题描述**：

方案第 4.2 节定义了 `sql.js` 方言辅助模块，其中的函数（`sql.now()`、`sql.jsonField()`、`sql.jsonFieldAs()` 等）需要根据当前数据库类型输出不同的 SQL 片段。例如 `sql.jsonField(col, path)` 在 SQLite 下输出 `json_extract(col, '$.path')`，在 KingbaseES 下输出 `col::jsonb->>'path'`。

但方案未定义 `sql.js` 如何获取当前数据库类型。`sql.js` 是一个纯函数模块，被路由层直接调用：

```javascript
const sql = require('../db/sql');
// 路由中使用：
const rows = await adapter.query(`SELECT ${sql.jsonField('result', 'risk_score')} FROM ...`, params);
```

`sql.js` 需要知道是 SQLite 还是 KingbaseES 才能生成正确方言。可能的方案包括：(a) 读取 `process.env.DB_TYPE` 环境变量；(b) 在 `initDatabase()` 时调用 `sql.setDialect(dbType)` 设置模块级状态；(c) 每个函数接受 `dbType` 参数。

方案未对此做出技术决策，这是一个遗漏的关键设计点——如果实现者选择了不合适的方式（如在每次函数调用时读取环境变量），可能导致性能问题或测试困难。

**改进建议**：

(1) 在第 4.2 节增加"方言感知机制"子节，明确 `sql.js` 的方言切换方式。推荐方案 (b)：`initDatabase()` 实例化 adapter 后调用 `sql.setDialect(dbType)` 设置模块级变量，`sql.js` 内部函数读取该变量。此方案避免每次函数调用时读环境变量，也便于单元测试（测试中可手动调用 `setDialect` 切换）。

(2) 在 `sql.js` 模块中增加方言未初始化时的防御性检查：如果方言未设置（`setDialect` 未被调用），抛出明确错误提示"sql.js dialect not initialized"而非生成错误 SQL。

### 问题 5：SqliteAdapter 同步异常转 Promise rejection 的实现策略文字描述有误导

- **所在位置**：方案第 3.3 节（SqliteAdapter 实现要点）
- **严重程度**：一般

**问题描述**：

方案第 3.3 节描述 SqliteAdapter 的异步封装策略为：

> "内部 `better-sqlite3` 是同步的，对外包裹 `Promise.resolve()` 即可"

这个描述在技术上不精确且有误导性。`better-sqlite3` 的同步方法（如 `db.prepare(sql).get(params)`）如果遇到 SQL 语法错误或约束违反，会**同步抛出异常**。而 `Promise.resolve(syncCall())` 中的 `syncCall()` 是在 Promise 构造之前执行的——如果它同步抛出异常，该异常不会被 Promise 捕获，而是作为未处理的同步异常向上传播，与 `async/await` 的错误处理模型不一致。

例如以下代码不会按预期工作：

```javascript
// 错误写法（方案文字可能引导的实现）
query(sql, params) {
  return Promise.resolve(db.prepare(sql).all(params)); // SQL错误时同步抛出，不返回rejected promise
}
```

正确的做法是使用 `async` 函数（其函数体会自动将同步异常转换为 rejected promise）：

```javascript
// 正确写法
async query(sql, params) {
  return db.prepare(sql).all(params); // 同步异常自动转为rejected promise
}
```

虽然有经验的开发者自然会使用 `async` 方法而非 `Promise.resolve()`，但方案作为技术指导文档，其文字描述应精确无误以避免误导初级实现者。

**改进建议**：

(1) 将第 3.3 节中"对外包裹 `Promise.resolve()` 即可"改为"SqliteAdapter 的 query/queryOne/execute 等方法声明为 `async` 函数。`async` 函数体会自动将 better-sqlite3 的同步异常转换为 rejected Promise，与 KingbaseAdapter 的异步错误模型一致，路由层的 `try/catch` 和 `await` 可统一处理。"

(2) 增加代码轮廓示例，明确 async 方法签名。

### 问题 7：Phase 2 数据迁移的停机时间估算缺失

- **所在位置**：方案第 12 节（数据迁移）、第 13.3 节（运维-停机时间）
- **严重程度**：一般

**问题描述**：

方案第 13.3 节提到"数据迁移期间需额外停机（取决于数据量）"，但没有提供任何估算方法或示例数字。对于一个需要规划生产切换的团队，缺少停机时间估算会导致：

1. 无法确定维护窗口大小
2. 无法评估对用户的影响
3. 无法与运维团队协调切换时间

迁移脚本涉及：读取 SQLite 所有 10 张表数据 -> 时区转换处理 -> 逐表写入 KingbaseES -> 验证（7 个维度） -> 序列重置。对于中小型数据量（如 < 10 万条记录），整个过程可能只需几分钟；但对于更大的数据集，可能需要数十分钟。

**改进建议**：

(1) 在第 12 节增加"停机时间估算"子节，提供基于数据量的估算公式或参考值。例如："假设 N 条记录，每条约 1KB，则迁移时间约 = SQLite读取时间 + 时区转换时间 + KingbaseES写入时间 + 验证时间。在典型开发机上（SSD，本地 KingbaseES），10 万条记录的迁移时间约 2-5 分钟。"

(2) 建议在 dry-run 阶段（12.1 节）实测迁移耗时，作为正式切换的参考。

(3) 讨论是否可以采用**在线迁移**策略（先迁移历史数据，再在切换前增量同步近期变更），以减少停机时间。

### 问题 8：迁移脚本中 SERIAL 序列名称的隐含假设未验证

- **所在位置**：方案第 12 节第 6 条（SERIAL 序列重置）
- **严重程度**：轻微

**问题描述**：

方案第 12 节的序列重置 SQL 使用了 `users_id_seq`、`doctor_information_id_seq` 等序列名称：

```sql
SELECT setval('users_id_seq', COALESCE((SELECT MAX(id) FROM users), 0) + 1, false);
```

这些序列名称是由 PostgreSQL/KingbaseES 在 `CREATE TABLE ... SERIAL PRIMARY KEY` 时自动生成的，命名规则为 `{tablename}_{columnname}_seq`。方案未提供验证这些序列名称正确性的方法。

如果 `init_kingbase.sql` 中的表名或主键列名与假设有任何偏差（例如历史原因导致某表主键列名为 `user_id` 而非 `id`），自动生成的序列名称会不同，导致 `setval` 调用失败。

**改进建议**：

(1) 在迁移脚本实现中，不硬编码序列名称，而是通过查询 `information_schema.columns` 动态获取每张表关联的序列名称：

```sql
SELECT pg_get_serial_sequence('users', 'id');  -- 返回正确的序列名称
```

(2) 在第 12 节第 6 条中增加注释，标注序列名称取决于 DDL 定义，建议在 dry-run 中验证序列名称正确性。

- **严重程度**：轻微

**问题描述**：

方案讨论了正常路径下的数据迁移和回退，但缺少以下异常场景的处理策略：

1. **迁移中途失败**：数据迁移脚本在写入第 5 张表后失败（如 KingbaseES 连接断开），前 4 张表已有部分数据。重新执行迁移脚本时如何处理这些部分数据？（方案 12 节提到"事务内执行"但跨表迁移通常不在单个事务内——10 张表的数据量可能很大，单事务不可行。）

2. **Phase 2 切换后发现数据正确性问题的回退**：方案 13.3 节回退方案是"将 `DB_TYPE` 改回 `sqlite` + 重启应用"，但如果在 KingbaseES 上已经运行了一段时间并产生了新数据，回退到 SQLite 会导致这段时间的数据丢失。方案提到了"逆向迁移脚本框架"但未讨论如何确定"需要回退"的触发条件。

**改进建议**：

(1) 在第 12 节增加"迁移异常处理"子节，明确：逐表迁移 + 每张表迁移后立即验证 + 支持断点续传（记录已迁移的表清单，重试时跳过）。

(2) 在第 13.3 节回退方案中增加回退决策触发条件（如：迁移后发现 > 1% 数据不一致、关键业务查询性能劣化 > 5 倍、P0 故障 > 2 次/天等）。

## 审查总结

**方案可实施性评估**：方案在整体架构设计（适配层模式、方言辅助函数、事务改造）方面可以直接指导实现。但在 Phase 0 的改动顺序（问题1）、时区脚本互斥关系（问题2）以及 sql.js 方言感知（问题3）这三个维度上存在工程实施障碍，需先澄清才能推进实现。

**关键遗漏的技术决策**：
1. Phase 0 过渡期 database.js 的向后兼容导出策略（问题1）
2. sql.js 的数据库类型感知机制（问题3）
3. 时区转换脚本的互斥执行策略（问题2）

**风险覆盖充分性**：风险表已覆盖 29 个风险项，较为全面。建议新增 2 个风险项：(1) Phase 0 路由文件改造原子性风险；(2) Phase 0/Phase 2 时区双重转换风险。

**建议处置优先级**：
1. **立即解决**（阻塞实现）：问题 1（Phase 0 增量改造可行性）、问题 2（时区双重转换冲突）
2. **实现前解决**（影响设计决策）：问题 3（sql.js 方言感知）、问题 4（Docker 镜像验证）、问题 6（测试策略）
3. **实现中完善**（不影响核心路径）：问题 5（文字精确性）、问题 7-10（辅助完善）
