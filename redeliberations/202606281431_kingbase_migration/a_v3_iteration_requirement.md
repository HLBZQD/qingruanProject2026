## 当前审查结果

本轮诊断报告（b_v2_diag_v1.md）经质询确认（LOCATED），共识别 2 个严重问题和 5 个一般性问题：

### 严重问题

**问题 1：`server.js` 未列入文件变更清单，且异步启动流程未给出完整代码轮廓**
- 严重程度：严重
- 所在位置：方案第 3.5 节（database.js 改造）、第 16 节（文件变更清单）
- 问题描述：方案要求 `initDatabase()` 改为 async，但第 16 节文件变更清单遗漏了 `server.js`。当前 `server.js` 中 `initDatabase()` 为同步调用，改为 `await initDatabase()` 后必须调整启动时序（`app.listen()` 需在数据库初始化完成后执行），方案未提供改造代码轮廓，也未讨论 `await` 在 CJS 模块根作用域的可用性问题。
- 改进建议：(1) 在第 16 节文件变更清单中增加 `server.js`，标注为"改造"；(2) 在第 3.5 节补充 `server.js` 的改造代码轮廓，明确 `await initDatabase()` 与 `app.listen()` 的时序关系；(3) 说明顶层 `await` 方案或 `.then()` 替代方案。

**问题 2：Phase 0 混合时间戳数据状态的处理策略缺失**
- 严重程度：严重
- 所在位置：方案第 4.2 节（`sql.now()` 与 UTC 存储决策）、第 6 节 Phase 0 验收标准
- 问题描述：Phase 0 切换到 `CURRENT_TIMESTAMP`（UTC 存储）后，旧数据（本地时间 UTC+8）与新数据（UTC）在 SQLite 中混合共存，导致：(a) 同一数据库内时间戳语义不一致，前端无法区分；(b) 时间范围查询出错（如 `punch.js` 第 125 行近 7 天查询）；(c) Dify AI 工作流影响未评估。混合状态贯穿 Phase 0 和 Phase 1，期间时间相关功能存在数据正确性风险。
- 改进建议：(1) 推荐在 Phase 0 改造 SQL 的同时，运行一次性 SQL 脚本将现有 SQLite 数据中所有 datetime 字段原地从本地时间转换为 UTC（减去 8 小时），使 Phase 0 启动后数据库立即进入全 UTC 状态；(2) 将此脚本作为 Phase 0 的前置步骤列入验收标准；(3) 在风险表中新增"Phase 0 混合时间戳数据一致性风险"条目。

### 一般问题

**问题 3：`/health` 端点改造与文件变更清单矛盾**
- 严重程度：一般
- 所在位置：方案第 13.2 节（监控与可观测性）与第 16 节（文件变更清单）
- 问题描述：方案 13.2 节建议 `/health` 端点返回数据库连接状态（调用 `adapter.healthCheck()`），但第 16 节标注 `server/routes/index.js` 为"零改动"。当前 `/health` 端点仅返回静态 JSON，未检查数据库连接。
- 改进建议：(1) 确认是否将 `/health` 改造纳入范围；(2) 若纳入，在第 16 节中更新 `server/routes/index.js` 标注；若暂不纳入，删除或标注 13.2 节相应描述。

**问题 4：`plan.js` 事务中批量 INSERT 的网络性能影响未评估**
- 严重程度：一般
- 所在位置：方案第 8.2 节（适配后的事务模式）、第 6 节 Phase 1 验收标准
- 问题描述：`plan.js` 的 `POST /generate` 在事务内用 `for` 循环逐条 INSERT。SQLite 中是进程内调用，KingbaseES 中每次都是网络往返，事务耗时可能从毫秒级升至秒级，导致锁持有时间延长、幂等锁不足、前端超时。
- 改进建议：(1) 在 Phase 1 性能基准测试中明确列为对比指标；(2) 考虑多行 INSERT 批量写入；(3) 在风险表中新增对应条目。

**问题 5：数据迁移验证策略深度不足**
- 严重程度：一般
- 所在位置：方案第 12 节（数据迁移）
- 问题描述：迁移验证仅"验证行数一致"不足以保证数据完整性。缺失的验证维度：内容正确性（时区转换抽样对比）、约束完整性（FK 有效性）、非空约束（NULL 比例）、JSON 字段有效性、迁移回滚演练。
- 改进建议：(1) 增加逐表行数对比、抽样逐列对比、FK 有效性检查、NULL 比例检查；(2) 增加 dry-run 说明（对 SQLite 副本执行）；(3) 提供逆向迁移脚本框架。

**问题 6：JSON 存储列类型在 `init_kingbase.sql` 中未明确决策**
- 严重程度：一般
- 所在位置：方案第 10.1 节（差异分析表）、第 10.2 节（对齐策略）、第 4.1 节（方言差异清单）
- 问题描述：`articles.tags`、`user_risk_info.result`、`user_risk_info.raw_input`、`admin_logs.operation_content`、`admin_logs.operation_result`、`life_advice.tags` 共 6 个 JSON 文本列，在 KingbaseES 中使用 TEXT 还是 JSONB 未决策。使用 TEXT 需运行时 `::jsonb` 转换；使用 JSONB 查询更快但需在迁移中验证 JSON 合法性。
- 改进建议：(1) 推荐生产环境使用 JSONB（配合 GIN 索引）；(2) 在迁移脚本中增加 JSON 合法性校验步骤。

**问题 7：KingbaseES 事务内 DDL 执行兼容性未验证**
- 严重程度：一般
- 所在位置：方案第 3.4.5 节（`init()` 方法）
- 问题描述：方案将 DDL（CREATE TABLE）包裹在事务中执行。虽然 PostgreSQL 支持事务内 DDL，但 KingbaseES V8R6 的事务内 DDL 行为需确认——若存在隐式提交 DDL，部分表 CREATE 无法回滚，幂等初始化在部分失败后留下中间状态。
- 改进建议：(1) 在 KingbaseES V8R6 上验证事务内 DDL 行为；(2) 将拆分 DDL/种子文件的备选方案提升为推荐方案；(3) 在风险表中新增对应条目。
