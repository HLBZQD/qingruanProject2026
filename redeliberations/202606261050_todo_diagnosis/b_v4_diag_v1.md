## 发现的质量问题

### 问题 1: Dify API Key 命名不匹配发现未作为独立问题纳入完整四维诊断框架

- **所在位置**：跨问题结构性分析 → 1 节「配置约定源冲突」（`a_v4_diag_v1.md:668-677`）
- **严重程度**：严重

报告在跨问题结构性分析中发现了一个关键运行时缺陷：`plan.js` 读取 `process.env.DIFY_PLAN_WORKFLOW_API_KEY`、`risk.js` 读取 `process.env.DIFY_RISK_WORKFLOW_API_KEY`，但 `.env` / `.env.example` 中定义的变量名为 `DIFY_PLAN_WORKFLOW_KEY` 和 `DIFY_RISK_WORKFLOW_KEY`（缺少 `_API` 中缀）。经代码验证：

- `plan.js:48` 使用 `DIFY_PLAN_WORKFLOW_API_KEY` → `.env` 定义为 `DIFY_PLAN_WORKFLOW_KEY` → **不匹配**，运行值为 `undefined`
- `risk.js:54` 使用 `DIFY_RISK_WORKFLOW_API_KEY` → `.env` 定义为 `DIFY_RISK_WORKFLOW_KEY` → **不匹配**，运行值为 `undefined`
- 当前 `DIFY_API_BASE=http://182.92.74.224/v1` 非空（非 Mock 模式），因此 `callWorkflowBlocking` 会发送 `Authorization: Bearer undefined` → Dify 返回 401 → 方案生成和风险预测**全部失败**

然而，这一发现仅被描述在跨问题结构性分析的对比表及附带段落中，**未作为独立问题纳入与原来 18 个问题同等规格的四维诊断**（根因分析、影响范围、严重性验证、关联性分析）。它与问题 2（`DB_PATH` vs `SQLITE_PATH`）存在本质差异——问题 2 中代码→.env→分批文档三者自洽（仅是设计文档不一致），而此处代码与配置文件**直接不匹配**，已在当前运行时产生功能故障。报告自身也指出"这是比问题 2 更严重的状况"（line 677），但未给予对应的深入诊断。

- **改进建议**：将此发现提升为独立问题（或在现有 18 个问题中明确新增一个），按四维框架完整展开诊断——根因（plan.js/risk.js 使用 `_API_KEY` 后缀，与 .env 的 `_KEY` 后缀不匹配）、影响范围（`POST /api/plan/generate`、`PUT /api/plan/adjust`、`POST /api/risk/predict` 在 DIFY_API_BASE 非空时全部失败）、严重性（严重——导致两个核心功能完全不可用，而非仅是约定不一致）、关联性（与问题 2/10 同属"分批文档命名约定偏离设计文档"模式，但后果严重程度不同）。

### 问题 2: 改进优先级表缺失 Dify API Key 命名不匹配的修复项

- **所在位置**：改进优先级建议 → P0/P1 表（`a_v4_diag_v1.md:754-768`）
- **严重程度**：严重

改进优先级建议章节对 18 个问题按 P0-P3 分级排序，但未将 Dify API Key 命名不匹配（`_API_KEY` vs `_KEY`）列为独立的修复项。该缺陷导致方案生成和风险预测两个核心功能在当前配置下实际不可用（或将在运维人员填入正确 API Key 后仍然不可用），属于功能阻断级缺陷（P0）。优先级表的缺失意味着修复者无法从优先级建议中识别此问题的紧迫性。

- **改进建议**：在 P0 优先级表中新增一项，修复内容为将 `plan.js`（4 处引用）和 `risk.js`（2 处引用）中的 `DIFY_PLAN_WORKFLOW_API_KEY` / `DIFY_RISK_WORKFLOW_API_KEY` 改为匹配 `.env` 的 `DIFY_PLAN_WORKFLOW_KEY` / `DIFY_RISK_WORKFLOW_KEY`（或更彻底的：统一以设计文档为准双向修正）。

### 问题 3: 跨问题配置约定源冲突对比表中的标注错误

- **所在位置**：跨问题结构性分析 → 1 节对比表第 4 行（`a_v4_diag_v1.md:660`）
- **严重程度**：中等

配置约定源冲突对比表中，"风险预测 API Key 变量名"一行的"代码/配置文件"列标注为 `DIFY_RISK_WORKFLOW_API_KEY`（**plan.js**），但实际使用该变量的代码在 `server/routes/risk.js:54`（而非 `plan.js`）。这是拷贝粘贴导致的标注错误——`plan.js` 使用的是 `DIFY_PLAN_WORKFLOW_API_KEY`（对应下一行的方案生成 API Key），两者指向不同的文件。

- **改进建议**：将第 4 行末尾的 `（plan.js）` 修正为 `（risk.js）`。

### 问题 4: 问题 2 的"完全自洽"声明与跨问题分析自身发现构成矛盾

- **所在位置**：问题 2 影响范围段落（`a_v4_diag_v1.md:89`）
- **严重程度**：一般

问题 2 的影响范围分析中称"代码、`.env`、`.env.example` 内部完全自洽，配置文件可正常加载"。该声明在问题 2 讨论的两个变量（`DB_PATH` 和 `DIFY_API_BASE`）范围内是正确的，但"完全"二字暗示代码库中所有环境变量均自洽。而同一份报告的后文（跨问题结构性分析 1 节，line 677）明确指出 Dify API Key 变量中 plan.js/risk.js 与 .env **不**自洽。同一份报告的前后两处存在语义张力——读者在读到 line 89 时可能产生"所有环境变量都没有运行时问题"的印象，读至 line 677 时才意识到存在真实的运行时不一致。

- **改进建议**：在问题 2 的"完全自洽"声明后增加范围限定，如"对于 DB_PATH 和 DIFY_API_BASE 这两个变量，代码、`.env`、`.env.example` 内部自洽"；或在声明后附加脚注，注明"此判断仅涵盖问题 2 讨论的两个环境变量，不适用于 Dify API Key 变量（见跨问题结构性分析 1 节）"。

## 整体评价

除上述问题外，产出在以下方面表现良好：
- 18 个问题均完成了根因分析（定位到代码行级偏差）、影响范围分析（指定端点/触发条件/后果）、严重性验证（与设计文档和分批文档交叉参照）、关联性分析
- 5 个跨问题结构性分析归纳准确，揭示了配置约定源冲突、模块顶层副作用反模式、Text2SQL 工具链依赖等系统性问题
- 4 轮迭代修订记录完整，每轮修订说明逐条回应了前序质询
- 代码引用具体可验证，环境变量三套命名约定的分析有明确的 `.env` / 代码 / 设计文档行号支撑

核心不足在于：报告中**最重要的新发现**（plan.js/risk.js 与 .env 的 Dify API Key 命名不匹配导致运行时功能故障）被埋没在跨问题结构性分析的附带段落中，既未纳入 18 问题的独立四维诊断框架，也未进入改进优先级表，且存在一处表内文件标注错误。这削弱了报告作为"修复者行动指南"的可操作性。

修复建议优先级：问题 1 和 2 应优先处理（将 Dify API Key 不匹配提升为独立诊断项并排入优先级表），问题 3 次之（修正标注错误），问题 4 和 5 为文档精度的打磨。
