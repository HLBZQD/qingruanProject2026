对 `a_v3_diag_v1.md`（后端代码问题诊断报告 v3）进行质量审查。审查侧重点：需求响应充分度、事实错误/逻辑矛盾、深度完整性和可操作性（根因深度、影响范围覆盖、严重性验证准确性、关联性揭示）。

## 质量问题

### 问题 1：问题 9 影响范围分析遗漏了幂等性守卫的交互效应

- **所在位置**：`a_v3_diag_v1.md:334-349`（问题 9 影响范围段落）
- **严重程度**：中等
- **问题描述**：影响范围分析指出 Dify 失败后用户"处于无活跃方案状态"，但未提及 `checkIdempotent()` 守卫（`server/routes/plan.js:13-21`）施加的 30 秒冷却锁。当 `POST /api/plan/generate` 因 Dify 调用失败而抛出异常后，`lastGenerateRequest.set(userId, now)` 已执行（line 19，先于事务），但 deactivate 已提交生效。用户此时同时面临两个障碍：(1) 无活跃方案；(2) 30 秒内无法重试生成新方案（返回 `409 CONFLICT`）。这是"数据丢失 + 操作封锁"的复合影响，比报告中描述的单一后果更为严重。
- **改进建议**：在问题 9 影响范围段落中补充幂等性守卫的交互效应，明确用户在该场景下不仅丢失活跃方案，还被 `checkIdempotent` 阻止立即重试。

### 问题 2：问题 5 未提及 `DIFY_SERVICE_API_KEY` 环境变量缺失

- **所在位置**：`a_v3_diag_v1.md:176-211`（问题 5 全部段落）
- **严重程度**：中等
- **问题描述**：报告正确指出 `difyAuth.js` 中间件缺失，但未指出该中间件的运行时依赖——`process.env.DIFY_SERVICE_API_KEY`——在当前 `.env`（8 行）和 `.env.example`（8 行）中均不存在。设计文档 6.3.2 节（`docs/2_detailed_design_v3.md:5329`）明确定义了 `DIFY_SERVICE_API_KEY=dify-service-key-xxxxxxxx`，但分批实现文档的 `.env` 模板（`3_backend_implementation_batches_v2.md:131-141`）省略了此项。这意味着即使实现了 `difyAuth.js`，中间件也会因 `process.env.DIFY_SERVICE_API_KEY` 为 `undefined` 而返回 500 内部错误（design doc 伪代码中对此情况的处理）。与问题 2（环境变量命名不一致）和问题 10（`JWT_EXPIRES_IN` 缺失）的模式类似，这是"分批文档省略了设计文档定义的配置项"的又一实例，但报告中未作为独立发现或关联分析进行标注。
- **改进建议**：(1) 在问题 5 的根因分析或关联性段落中补充说明 `DIFY_SERVICE_API_KEY` 环境变量缺失；(2) 在跨问题结构性分析 1b 节的"命名分歧"类别中增加此项（或作为新的"配置项省略"子类），将受影响问题从 7 个扩展为 8 个（问题 2、5、10 共享此模式）。

### 问题 3：问题 15 的根因分析未引用分批文档的明确指示

- **所在位置**：`a_v3_diag_v1.md:514-538`（问题 15 根因分析段落）
- **严重程度**：一般
- **问题描述**：报告将对话历史会话列表的桩实现归因为"(1) Dify Conversations API 尚未在初期版本中提供；(2) 分批实现的优先级排序——会话历史是辅助功能"。但分批实现文档（`3_backend_implementation_batches_v2.md:766` 和 `789`）给出了明确指示：
  - `GET /api/chat/doctor/:id/conversations`："历史会话列表第一版可先返回空数组"
  - `GET /api/assistant/conversations`："第一版可返回空数组"
  
  这表明桩实现并非"优先级排序导致未实现"，而是**分批文档的明确设计要求**——第一版就是返回空数组。报告的根因分析用推测性语言掩盖了分批文档中的显式设计决策，归类不够精确。若引用分批文档原文，此问题的定性应为"符合分批计划的第一版实现"而非"功能缺失"，修复时机应推迟至后续批次而非 P3。
- **改进建议**：(1) 修改问题 15 的根因分析，引用分批文档 line 766/789 的明确指示；(2) 重新评估严重性——当前框架下此问题更接近"按计划实现"而非"功能缺失"，修复优先级建议可保留 P3 但应标注"符合分批计划"。

### 问题 4：跨问题结构性分析遗漏了 Dify API Key 变量命名的系统性偏差

- **所在位置**：`a_v3_diag_v1.md:640-668`（配置约定源冲突和偏差性质分析）
- **严重程度**：一般
- **问题描述**：报告在"配置约定源冲突"分析中仅覆盖了 `DB_PATH`/`SQLITE_PATH` 和 `DIFY_API_BASE`/`DIFY_API_BASE_URL` 两个命名对（加上 `JWT_EXPIRES_IN` 缺失），但设计文档 vs 分批文档的变量命名分歧远不止于此。设计文档 `2_detailed_design_v3.md:5330-5335` 定义的 Dify API Key 变量使用 `_API_KEY` 后缀：
  - `DIFY_ASSISTANT_API_KEY`
  - `DIFY_RISK_WORKFLOW_API_KEY`
  - `DIFY_PLAN_WORKFLOW_API_KEY`
  - `DIFY_ARTICLE_WORKFLOW_API_KEY`
  
  而分批文档和代码实际使用：
  - `DIFY_ASSISTANT_APP_KEY`（后缀 `_APP_KEY`）
  - `DIFY_RISK_WORKFLOW_KEY`（后缀 `_KEY`，无 `_API_`）
  - `DIFY_PLAN_WORKFLOW_KEY`
  - `DIFY_ARTICLE_WORKFLOW_KEY`

  这 4 个变量名的分歧模式与问题 2 同源，共同构成"分批文档系统性偏离设计文档的变量命名约定"这一更广泛的问题。报告未揭示此范围，减弱了跨问题分析的覆盖广度。
- **改进建议**：在跨问题结构性分析 1 节（配置约定源冲突）的对比表中，补充 Dify API Key 变量命名分歧的各行，或在 1b 节"命名分歧"类别中增加说明该模式不仅限于问题 2 覆盖的两个变量。

### 问题 5：问题 51 （`difyAuth`）的依赖分析未区分"代码实现依赖"和"平台能力依赖"

- **所在位置**：`a_v3_diag_v1.md:210-211`（问题 5 关联性段落末尾）
- **严重程度**：一般
- **问题描述**：问题 5 的关联性分析正确指出"依赖 Dify 平台能力验证——设计文档 5.5.1 节定义的门禁验证任务尚未执行"，并将此区分为独立的依赖风险。但问题 4 的关联性分析（line 172）已建立了"代码实现层面"与"功能层面"的区分框架——这一区分在问题 5 的关联性分析中同样适用却未复用。`difyAuth` 的**代码实现**不依赖门禁验证结果——常量时间比较 `req.body.api_key === process.env.DIFY_SERVICE_API_KEY` 和提取 `req.body.user_id` 的逻辑不依赖 Dify 的 `{{user}}` 变量透传能力。门禁验证未通过仅影响"运行时 Dify Agent 回调链路能否正常工作"，不影响中间件的编写和单元测试。两处以不同框架描述类似性质的依赖，缺少一致性。
- **改进建议**：在问题 5 的关联性分析中复用问题 4 建立的"代码实现层面 vs 功能/运行时层面"区分框架，明确：(1) `difyAuth.js` 代码实现不依赖于门禁验证结果；(2) 门禁验证失败仅影响运行时 Dify Agent 回调链路的有效性，其备选方案（session_id 映射表）会影响 `user_id` 的提取逻辑。

### 问题 6：改进优先级表中 P1 级别排列了 3 个不同性质的问题，合并建议可能导致执行混淆

- **所在位置**：`a_v3_diag_v1.md:736-742`（P1 优先级建议表格）
- **严重程度**：轻微
- **问题描述**：P1 行将"问题 2 + 10（环境变量 + JWT 过期）"合并为一个任务，理由是"同源配置问题，可批量处理"。但问题 2 的重命名修复（修改 `database.js:10`、`difyService.js:85`、`sseProxy.js:10` 和 `.env.example`/`.env` 中的变量名）与问题 10 的过期时间修复（将 `routes/auth.js:35,74` 硬编码 `'7d'` 改为读取 `process.env.JWT_EXPIRES_IN`）属于不同性质的变更——前者是重命名（代码和配置同步修改），后者是重构（从硬编码变为环境变量读取 + `.env` 新增条目）。此外，问题 4（P2 优先级 6）的跨问题分析已指出问题 2 的 Dify 变量命名分歧可能进一步扩大。将不同性质的变更捆绑为一个 P1 任务，可能导致实际修复时被拆分为多次提交。
- **改进建议**：将问题 2 和问题 10 拆分为独立的 P1 行，分别标注修复性质和涉及文件。或在表格中增加"修复方式"列说明差异。
